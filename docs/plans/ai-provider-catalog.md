# AI Provider Catalog Design

**Status:** implementation slice in progress
**Last updated:** 2026-06-01
**Owner:** product + platform

## Summary

This document defines the OSS AI Provider layer for Clawdi. The core
decision is that Clawdi does not proxy user BYOK model traffic. User-owned
OpenAI, Anthropic, OpenRouter, Gemini, Mistral, and compatible endpoint keys
are resolved by the local runtime or hosted agent service, and that runtime
calls the provider directly. Clawdi Cloud is the OSS control plane in this
repository: dashboard, backend, CLI contracts, storage, and docs. It may
collect provider settings during onboarding, encrypt and store secrets, return
OAuth authorization links, inject deployment secrets, and distribute a provider
catalog, but it is not in the model request path for BYOK traffic.

The source of truth is a portable Provider Catalog. Codex, OpenClaw, Hermes,
the CLI, local self-hosted runtimes, and the hosted agent service consume the
same catalog and receive engine-specific projected config generated by the
Clawdi CLI. Native runtime config and Codex profile files are disposable
projections, not the user-owned source of truth. API keys, OAuth connections,
and whole tool credential profiles are managed as the `auth` part of an AI
Provider. The backend may store encrypted auth payloads in separate internal
tables, but the product object remains AI Provider.

This document distinguishes two paths:

1. **Model traffic path**: runtime to model provider. User BYOK model traffic
   must not pass through Clawdi Cloud.
2. **Secret resolution path**: runtime or CLI to a secret source. `env:` refs
   resolve locally; `clawdi://` refs may call Clawdi Vault before the runtime
   calls the model provider directly.

This direction supersedes the BYOK parts of
`docs/scenarios/scenario-5-gateway-unified-api.md`. A Clawdi-managed Gateway
can still exist for Clawdi-funded or platform-managed traffic, but it must not
be the path for user-owned BYOK provider calls.

## Reasonableness Assessment

The request is reasonable and should become the product direction. It fixes
three current risks:

1. It removes duplicated provider configuration from OpenClaw, Hermes, Cloud,
   and ad hoc CLI flows.
2. It makes the security boundary explicit: BYOK calls go directly from runtime
   to provider, while Clawdi only handles metadata, secret custody, and config
   materialization.
3. It creates a backup/restore unit that can move across local, self-hosted,
   and hosted agent service environments without scattering raw API keys into
   runtime config files.

There are four required refinements:

1. The schema needs an explicit provider protocol or API mode. Capabilities
   such as `chat`, `responses`, and `tools` are not enough to generate stable
   OpenClaw or Hermes config for OpenAI-compatible, Anthropic-compatible, and
   custom endpoints.
2. The CLI may provide base URL and env var suggestions, but it should not own
   a long-lived default model table. The user's catalog owns `default_model`.
3. Secret backup must be a separate encrypted bundle. Default backup exports
   provider metadata and secret references only.
4. Provider auth management must be broader than API keys. Existing local agent
   credential profiles such as Codex `~/.codex/auth.json` and Claude Code
   credentials should migrate under AI Provider auth as first-class encrypted
   payloads or refs, not become a separate user-facing product.
5. Clawdi Cloud must treat provider projection as OSS CLI behavior. The hosted
   agent service consumes the catalog and invokes CLI materialization; it
   should not duplicate OpenClaw/Hermes config formats, provider header
   quirks, or runtime API-mode logic.

## Review Corrections

The first design pass intentionally marked OpenClaw and Hermes output as
illustrative, but it still treated several unverified assumptions as if they
were likely contracts. The implementation plan must correct that:

1. Projection work is blocked on pinned OpenClaw/Hermes provider-config
   fixtures. Local source review has identified likely contracts, but golden
   tests must lock verified runtime fixtures, not guessed shapes.
2. AI Provider secret handling must reuse Clawdi's existing
   `secret-references.ts` and Vault resolver instead of creating a parallel
   `clawdi://` parser.
3. Audit copy must distinguish direct model traffic from secret resolution.
   `clawdi://` refs may call Clawdi Vault even though BYOK model calls bypass
   Clawdi Cloud.
4. Runtime adapters must not treat "render" as "edit the user's primary
   runtime config." Rendering is pure projection, writing materializes
   Clawdi-owned files, and activation is a separate runtime-specific step.
5. Hermes native config writes, when explicitly requested, must be structured
   merges that preserve `mcp_servers.clawdi` from `clawdi setup`.
6. Drift metadata must be sidecar by default; native config files should not
   receive Clawdi metadata keys unless the pinned runtime schema allows them.
7. Gateway is represented only as an explicit Clawdi-managed provider. It is
   not an automatic fallback that rewrites user BYOK providers.

## Implementation Readiness Notes

Repository and adjacent-runtime review produced these constraints:

1. The current Clawdi CLI already has the secret-reference foundation needed
   for this work: `clawdi://` parsing, Vault resolution, preview, bulk
   resolution, and `clawdi run` env injection live in
   `packages/cli/src/lib/secret-references.ts` and
   `packages/cli/src/commands/run.ts`.
2. The current Clawdi CLI does not appear to have a general `clawdi backup` /
   `clawdi restore` command. Provider backup is therefore either a new product
   surface, or v1 should treat `ai-provider export/import` as the provider-only
   backup path until a global backup command exists.
3. `clawdi setup` already writes Hermes MCP registration into
   `~/.hermes/config.yaml`. Hermes projection must not replace this file, and
   native patch activation must preserve unrelated YAML sections such as
   `mcp_servers`.
4. Local OpenClaw source review shows support for `$include`,
   `OPENCLAW_CONFIG_PATH`, `OPENCLAW_INCLUDE_ROOTS`,
   `models.providers.<id>`, env-shaped `SecretInput` objects, and OpenClaw
   model API labels such as `openai-completions`, `openai-responses`,
   `anthropic-messages`, and `google-generative-ai`.
5. Local OpenClaw also exposes non-interactive config commands such as
   `openclaw config set`, `openclaw config patch`, `--batch-json`,
   `--merge`, and `--dry-run`. When a runtime CLI has a stable structured
   patch contract, the Clawdi adapter should prefer that CLI over editing
   native config files itself.
6. Local Hermes source and docs review shows `config.yaml` is resolved from
   `HERMES_HOME` and shows support for keyed `providers`,
   legacy `custom_providers`, `key_env` / `api_key_env`, and API modes
   `chat_completions`, `codex_responses`, and `anthropic_messages`.
7. Hermes exposes `hermes config set`. V1 may use it only under explicit
   `runtime render --write --activate`, after tests prove it preserves
   unrelated config such as `mcp_servers.clawdi`. Plain `--write` remains
   managed-file materialization and does not edit `~/.hermes/config.yaml`.
8. Official Codex config docs and local `codex-cli 0.135.0` review show the
   current Codex config contract: Codex loads user config from
   `$CODEX_HOME/config.toml`, supports selected profile files named
   `$CODEX_HOME/<profile>.config.toml` through `codex --profile`, supports
   one-off `-c key=value` overrides, and intentionally prevents project
   `.codex/config.toml` from overriding provider/auth/profile keys.
9. Codex v1 activation should therefore use a Clawdi-owned Codex profile file,
   not the user's primary `config.toml` and not project config. Support should
   be pinned to the feature contract, not one narrow package version:
   `@openai/codex <1.0.0` is eligible when the installed version supports
   profile config, `model_providers`, and `wire_api = "responses"`.
10. The findings above are implementation evidence, not permanent product
   contracts. Phase 0.5 must copy representative runtime config examples into
   Clawdi fixtures for the exact pinned versions that Clawdi supports.
11. The current CLI and backend already have local agent credential profile
   import/materialize support under `clawdi agent credentials` and
   `/api/vault/credential-profiles`. That implementation should be reused as
   the migration seed for provider auth instead of duplicated.

## Product Goals

The product goal is to make Clawdi the portable AI Provider control layer for
agent runtimes. Users should configure provider access once, keep stable
provider IDs across machines and runtimes, and move between local, self-hosted,
and hosted environments without reauthoring OpenClaw or Hermes config by hand.

The user-visible outcomes:

1. **Portability**: a provider such as `openai-main` follows the user across
   machines, backup/restore, local runtime, self-hosted runtime, and hosted
   agent service runtime.
2. **Runtime neutrality**: Codex, OpenClaw, Hermes, and future engines consume the
   same provider catalog. Engine-native config is generated output.
3. **Direct BYOK privacy**: when the user provides their own provider key,
   model requests go from runtime to provider. Clawdi does not proxy or record
   request bodies. Secret resolution may still use Clawdi Vault when the
   provider auth ref is `clawdi://...`.
4. **Secret minimization**: provider catalogs, runtime configs, backups, logs,
   and diagnostics carry secret refs or availability status, not plaintext
   API keys.
5. **Auth portability**: API keys, OAuth connections, and agent login profiles
   are managed as AI Provider auth instead of scattered per-runtime files.
6. **Recoverability**: a user can back up provider metadata by default and can
   optionally make an explicitly encrypted secret backup.
7. **Cloud simplicity**: Clawdi Cloud can collect provider settings, return
   OAuth authorization links, and inject secrets, but provider-specific
   runtime details stay in the OSS CLI.
8. **Auditability**: `validate`, `test`, and `runtime inspect` make it clear
   which provider will be used, which secret ref is required, and whether the
   request path is direct or Clawdi-managed.

Product success criteria:

1. A user adds `openai-main` once and renders Codex, OpenClaw, and Hermes
   config from the same catalog.
2. A user migrates an existing Hermes or OpenClaw provider config into the
   catalog, then regenerates equivalent runtime config.
3. A user rotates `OPENAI_API_KEY` without editing Codex, OpenClaw, or Hermes
   config.
4. A default backup restores provider metadata and secret refs on a new
   machine without exposing plaintext keys.
5. An encrypted secret backup cannot be created without an explicit passphrase
   or public-key recipient.
6. A user can import or connect Codex/Claude credentials, store them centrally,
   and materialize them on a new machine without hand-copying auth files.
7. A hosted agent runtime receives a provider catalog plus auth metadata and
   secret env/materialization inputs, and can start without Clawdi Cloud
   generating engine-native provider config.
8. A BYOK test call and runtime model call can be inspected and shown to target
   the provider base URL as the model endpoint, not a Clawdi Cloud model proxy.

UX principles:

1. The primary user concept is "AI Provider", not "BYOK config".
2. Provider ID is the stable handle users and runtimes reference.
3. The happy path should be one provider add, one direct test, one runtime
   render.
4. Every command that could activate or patch runtime config should support
   dry-run and show a redacted diff before writing.
5. Missing secrets should be actionable: name the missing ref and where the
   user can provide it, but never print stored secret values.

## Implementation Goals

1. Define one AI Provider entity and one Provider Catalog file format.
2. Let users refer to providers by stable user-owned IDs such as
   `openai-main`.
3. Keep secrets out of Codex/OpenClaw/Hermes config by default through secret
   refs or env names.
4. Support `env:` and existing `clawdi://` Vault secret refs in v1, then add
   keychain and 1Password through the same resolver interface.
5. Generate deterministic runtime projections for Codex, OpenClaw, and Hermes.
6. Include provider metadata in Clawdi backup and restore.
7. Allow explicit encrypted secret backup without ever placing plaintext keys
   in default backup archives.
8. Centralize API keys, OAuth connections, and agent credential profiles under
   AI Provider auth, while keeping provider metadata separate from encrypted
   token payloads internally.
9. Document that BYOK model requests are direct provider calls, not Clawdi
   proxy calls.

## Non-goals

1. No Clawdi proxying for user BYOK model traffic.
2. No plaintext API keys in the Provider Catalog.
3. No default plaintext secret export in `clawdi backup`.
4. No Cloud-maintained OpenClaw or Hermes provider config generator.
5. No Cloud-maintained provider default model table.
6. No broad home-directory scan to discover API keys.
7. No automatic shell startup file edits to install restored env vars.
8. No storage of whole OAuth tokens or local agent auth files inside Provider
   Catalog metadata. Those belong in the encrypted provider-auth payload store
   and are referenced by provider auth metadata.

## Terminology

- **AI Provider**: a user-defined model provider endpoint plus auth reference,
  default model, and declared runtime capabilities.
- **Model Provider**: acceptable user-facing synonym when the surface is
  specifically about model calls.
- **Provider Catalog**: the portable JSON document containing AI Provider
  definitions.
- **Provider Auth**: the auth portion of an AI Provider. It may be an env/Vault
  ref, encrypted API key payload, OAuth connection, or local agent credential
  profile.
- **Credential Profile**: a tool-specific local login artifact, for example
  Codex `~/.codex/auth.json` or Claude Code credentials, attached to an AI
  Provider auth object and materialized only by explicit adapter commands.
- **OAuth Connection**: provider auth acquired through an official provider or
  tool OAuth flow, including device-code, PKCE loopback, or manual paste
  completion when the provider supports it.
- **Secret Ref**: a pointer to a secret source, for example
  `env:OPENAI_API_KEY`.
- **Projection**: engine-specific generated runtime config for OpenClaw,
  Hermes, or another runtime.
- **BYOK**: an auth source pattern, not the product object name.

## User Flows

### Flow 1: New Local User Adds A Provider

The user has an OpenAI key in their shell and wants Hermes or OpenClaw to use
it without hand-editing native config.

```bash
export OPENAI_API_KEY=...

clawdi ai-provider add openai-main \
  --type openai \
  --base-url https://api.openai.com/v1 \
  --default-model gpt-5.2 \
  --api-mode openai_responses \
  --auth env:OPENAI_API_KEY

clawdi ai-provider validate openai-main
clawdi ai-provider test openai-main
clawdi runtime render --engine hermes --write --activate
```

Expected product behavior:

1. `add` writes a catalog entry with `auth.ref = env:OPENAI_API_KEY`.
2. `validate` checks schema and secret ref shape without making a network call.
3. `test` resolves `OPENAI_API_KEY` from the local environment and calls
   `https://api.openai.com/v1` directly.
4. `runtime render --write` materializes Clawdi-owned projection files.
   `--activate` shows a redacted activation plan. If the verified Hermes
   version has no include/profile mechanism that can layer provider config,
   local activation requires explicit native-patch consent and updates only
   provider/model-owned sections.
5. Hermes starts with the projected provider config and calls OpenAI directly.

### Flow 2: Existing Hermes Or OpenClaw User Migrates

The user already configured providers in native runtime config and wants Clawdi
to become the source of truth.

```bash
clawdi ai-provider import --from-hermes ~/.hermes/config.yaml
clawdi ai-provider validate
clawdi runtime inspect
clawdi runtime render --engine openclaw --write --activate
```

Expected product behavior:

1. Import reads Hermes `providers`, legacy `custom_providers`, and `model`
   defaults.
2. Env markers such as `key_env: OPENAI_API_KEY` become
   `auth.ref = env:OPENAI_API_KEY`.
3. Plaintext `api_key` fields are not imported silently. The CLI asks for an
   explicit secret target or skips the plaintext secret while preserving
   provider metadata.
4. Provider IDs are derived from existing provider keys where possible.
5. The user can render and activate OpenClaw config from the imported catalog
   without retyping provider details. OpenClaw activation should use
   `OPENCLAW_CONFIG_PATH` or `$include` before any native config patch is
   considered.

### Flow 3: User Rotates A Provider Key

The user rotates their OpenAI key at the provider dashboard.

```bash
export OPENAI_API_KEY=...
clawdi ai-provider test openai-main
clawdi runtime inspect
```

Expected product behavior:

1. The catalog does not change because it references `env:OPENAI_API_KEY`.
2. Engine-native config does not need to change.
3. `test` verifies the new key by calling the provider directly.
4. `runtime inspect` reports that `openai-main` has an available secret ref.

### Flow 4: Backup And Restore To A New Machine

The user wants provider definitions on a new machine without accidentally
exporting API keys.

```bash
# Future global backup surface:
clawdi backup --out clawdi-backup.zip

# On the new machine:
clawdi restore clawdi-backup.zip
clawdi ai-provider validate
clawdi runtime inspect

# Provider-only v1 fallback if global backup is not implemented yet:
clawdi ai-provider export --out ai-providers.json
clawdi ai-provider import ai-providers.json
```

Expected product behavior:

1. The default backup includes provider metadata and secret refs.
2. The default backup does not include plaintext secret values.
3. Restore recreates the catalog.
4. `validate` passes schema checks but reports missing secret refs until the
   user provides them or logs in for Vault-backed refs.
5. Once the user sets the env vars, `test` and `runtime render` work normally.

Optional encrypted secret backup:

```bash
clawdi backup --include-secrets --secret-passphrase --out clawdi-backup.zip
clawdi restore clawdi-backup.zip --secrets-target env-file --out ~/.clawdi/providers.env
```

Expected product behavior:

1. The command refuses to include secrets without encryption.
2. Restore writes owner-only secret material to the selected target.
3. Restore does not edit shell startup files.

### Flow 5: Hosted Agent Service Uses The Same Catalog

The user deploys a hosted agent and chooses their own provider key during
Cloud onboarding.

Expected product behavior:

1. Dashboard collects provider type, label, base URL, default model, and API
   key.
2. Clawdi Cloud encrypts the key and stores provider metadata.
3. The hosted agent service injects the key as an env var such as
   `OPENAI_API_KEY`.
4. Clawdi Cloud sends the Provider Catalog to the hosted agent service.
5. The hosted agent service runs Clawdi CLI materialization to generate
   OpenClaw or Hermes config.
6. OpenClaw or Hermes calls the provider API directly.
7. Clawdi Cloud can show setup status and validation results, but it does not
   proxy BYOK model traffic or maintain engine-specific provider config code.

### Flow 6: Self-hosted Or Team Environment Shares Definitions

A team wants the same provider IDs across self-hosted agents, but each runtime
has its own secret source.

```bash
clawdi ai-provider export --out providers.json
clawdi ai-provider import providers.json
clawdi ai-provider edit openai-main --auth env:TEAM_OPENAI_API_KEY
clawdi runtime render --engine openclaw --write --activate
```

Expected product behavior:

1. Provider IDs and non-secret metadata are portable.
2. Secret refs can be remapped per environment.
3. Scripts and runtime policies can keep referencing `openai-main`.
4. No exported catalog contains a plaintext team API key by default.

### Flow 7: User Configures Multiple Providers

The user wants OpenRouter for general routing and Anthropic direct for a
specific model.

```bash
clawdi ai-provider add openrouter-main \
  --type openrouter \
  --base-url https://openrouter.ai/api/v1 \
  --default-model anthropic/claude-opus-4.6 \
  --api-mode openai_chat \
  --auth env:OPENROUTER_API_KEY

clawdi ai-provider add anthropic-direct \
  --type anthropic \
  --base-url https://api.anthropic.com \
  --default-model claude-opus-4-6 \
  --api-mode anthropic_messages \
  --auth env:ANTHROPIC_API_KEY

clawdi ai-provider list
clawdi runtime inspect
```

Expected product behavior:

1. The catalog can hold multiple providers with stable IDs.
2. Defaults decide the normal runtime path.
3. Runtime projection can include all providers or only selected providers,
   depending on engine support and CLI flags.
4. Inspect output shows which provider is default and which secret refs are
   missing or available.

### Flow 8: User Configures A Local No-auth Provider

The user runs a local OpenAI-compatible endpoint such as LM Studio, Ollama, or
vLLM and does not need an API key.

```bash
clawdi ai-provider add lmstudio-local \
  --type custom_openai_compatible \
  --base-url http://127.0.0.1:1234/v1 \
  --default-model local-model \
  --api-mode openai_chat \
  --auth none

clawdi ai-provider test lmstudio-local
clawdi runtime render --engine hermes --write --activate
```

Expected product behavior:

1. `auth: none` is allowed for local or otherwise explicitly trusted endpoints.
2. `validate` warns when `auth: none` is used with public network origins unless
   the user explicitly allows it.
3. Projection emits no API key field or emits the runtime's documented no-auth
   marker.
4. Model traffic still goes directly from runtime to the local endpoint.

### Flow 9: User Uses A Vault-backed Provider Secret

The user wants the provider catalog to reference a Clawdi Vault secret instead
of a shell-managed env var.

```bash
clawdi vault set default/openai/api_key --prompt

clawdi ai-provider add openai-main \
  --type openai \
  --base-url https://api.openai.com/v1 \
  --default-model gpt-5.2 \
  --api-mode openai_responses \
  --auth clawdi://default/openai/api_key \
  --runtime-env OPENAI_API_KEY

printf 'OPENAI_API_KEY=clawdi://default/openai/api_key\n' > .env.clawdi
clawdi runtime render --engine hermes --write --activate
clawdi run --env-file .env.clawdi -- hermes
```

Expected product behavior:

1. The catalog stores `auth.ref = clawdi://default/openai/api_key`.
2. `runtime render` emits engine config that expects `OPENAI_API_KEY` or the
   runtime's verified secret-ref equivalent.
3. `clawdi run` resolves the `clawdi://` ref through Clawdi Vault and injects
   `OPENAI_API_KEY` into the Hermes process.
4. Hermes calls `https://api.openai.com/v1` directly after startup.
5. Audit output distinguishes the Vault resolution request from the direct
   model provider request.

### Flow 10: User Connects Codex Or Claude Provider Auth

The user wants Clawdi to centrally store and restore a runtime login, not just
an API key.

```bash
clawdi ai-provider connect openai-codex --method oauth --tool codex --profile default
clawdi ai-provider connect claude-code-main --method oauth --tool claude-code --profile default
clawdi ai-provider list
clawdi ai-provider materialize-auth openai-codex
```

Expected product behavior:

1. `connect` starts the verified provider/tool OAuth flow and prints a login
   URL, device code, or loopback URL according to that flow. CLI defaults to a
   loopback callback when the provider allows native-app loopback redirects.
2. If browser loopback is unavailable, the CLI supports manual completion with
   a loopback redirect such as `http://127.0.0.1:14565/callback`; the user can
   paste the failed browser callback into `complete-oauth --redirect-url <url>`
   or pass `--code/--state`.
3. The backend exchanges the code only through official provider/tool
   contracts. The CLI forwards `code`, `state`, and redirect URI but does not
   receive provider tokens. Neither side may scrape browser cookies or
   reverse-engineer private token stores.
4. The resulting encrypted provider-auth payload is account-global by default
   and stores redacted metadata such as provider, profile, scopes, expiry, and
   last test status.
5. `materialize-auth` writes only the tool's expected local credential artifact,
   using the verified adapter for Codex, Claude Code, Hermes, or another
   runtime. It does not modify shell startup files or Provider Catalog
   metadata.
6. AI Provider entries reference that auth payload through `auth.payload_ref`
   when a runtime provider uses that login state.

V1 key-env runtime projections only support auth forms that can be expressed
as a runtime environment variable or no-auth local endpoint. `agent_profile`
and `oauth_profile` entries are centrally managed provider auth, but they must
be materialized through a verified native tool adapter until a target runtime
has a pinned credential-profile config contract. Codex is the first exception:
`agent:codex/<profile>` can project to Codex native auth under the pinned
Codex profile contract. Other runtimes must not project auth profiles as
`key_env` just because a `runtime_env_name` is present.

### Flow 11: User Activates A Codex Provider Profile

The user wants Codex to use the same Provider Catalog without Clawdi editing
their primary Codex config.

```bash
clawdi ai-provider add openai-main \
  --type openai \
  --base-url https://api.openai.com/v1 \
  --default-model gpt-5.2 \
  --api-mode openai_responses \
  --auth env:OPENAI_API_KEY

clawdi runtime render --engine codex --write --activate
codex --profile clawdi-ai-provider
```

Expected product behavior:

1. `runtime render` emits deterministic Codex TOML with
   `wire_api = "responses"` and an `env_key`, or uses Codex native OpenAI auth
   when the provider auth is `agent:codex/<profile>`.
2. `--write` stores the generated file under
   `$CLAWDI_HOME/runtime/codex/`.
3. `--activate` writes only
   `$CODEX_HOME/clawdi-ai-provider.config.toml` and keeps
   `$CODEX_HOME/config.toml` unchanged.
4. The user starts Codex with `codex --profile clawdi-ai-provider`.
5. Codex calls the provider API directly; Clawdi is not in the model request
   path.

### Failure States

Common failure states should be explicit and recoverable:

1. **No catalog**: `runtime render` explains that no Provider Catalog exists
   and suggests `clawdi ai-provider add`.
2. **Missing env ref**: `test` and `runtime inspect` show
   `env:OPENAI_API_KEY` is missing and do not make a network call.
3. **Bad base URL**: `validate` fails before writing runtime config.
4. **Unsupported projection**: `runtime render --engine hermes` reports which
   provider field cannot be represented in Hermes config.
5. **Native config drift**: `runtime inspect` identifies generated files that
   differ from the current catalog projection.
6. **Import conflict**: import refuses to overwrite an existing provider ID
   without an explicit replace or rename.
7. **Provider test failure**: `test` reports provider host, status family, and
   redacted error class, but does not print request bodies or keys.

## Catalog Storage

The OSS CLI stores the local Provider Catalog at:

```text
$CLAWDI_HOME/ai-providers/catalog.json
```

`$CLAWDI_HOME` follows the existing Clawdi config home behavior and defaults to
`~/.clawdi`. The catalog file must be written owner-only (`0600`) because it
contains endpoint metadata and secret references, even though it does not
contain plaintext secrets.

Runtime projection output is separate:

```text
$CLAWDI_HOME/runtime/openclaw/
$CLAWDI_HOME/runtime/hermes/
```

Projection files are materialized as Clawdi-owned artifacts. They are connected
to an engine only through an explicit activation step. The catalog is still
authoritative.

## Catalog Schema

The catalog is JSON for portability and deterministic golden tests.

```json
{
  "schema_version": 1,
  "providers": [
    {
      "id": "openai-main",
      "type": "openai",
      "label": "OpenAI main",
      "base_url": "https://api.openai.com/v1",
      "default_model": "gpt-5.2",
      "api_mode": "openai_responses",
      "auth": {
        "type": "secret_ref",
        "ref": "env:OPENAI_API_KEY"
      },
      "capabilities": {
        "chat": true,
        "responses": true,
        "tools": true,
        "vision": true
      }
    }
  ],
  "defaults": {
    "chat_provider_id": "openai-main"
  }
}
```

`auth.type = "secret_ref"` is for lightweight env/Vault pointers.
`auth.type = "api_key"` is for Clawdi-managed API key payloads or refs.
`auth.type = "oauth_profile"` is for official provider/tool OAuth connections.
`auth.type = "agent_profile"` is for a tool-specific auth profile such as
Codex or Claude Code. Provider metadata never stores raw token payloads.

Provider IDs are user-owned stable references. They should match:

```text
^[a-z][a-z0-9._-]{1,62}$
```

Provider auth profile identifiers are separate from the slash-delimited
`<provider-or-tool>/<profile>` CLI syntax. The individual provider/tool and
profile segments should match:

```text
^[a-z][a-z0-9._-]{0,119}$
```

Managed provider-auth payload refs are internal pointers and must match the
owning provider ID:

```text
ai-provider-auth://<provider-id>/<profile>
```

Provider types for v1:

```text
openai
anthropic
openrouter
gemini
mistral
custom_openai_compatible
```

API modes for v1:

```text
openai_chat
openai_responses
anthropic_messages
google_generate_content
```

`api_mode` is optional only when `type` has one unambiguous default. It should
be required for `custom_openai_compatible`, and the CLI should write it during
import/migration when it can infer the runtime protocol from an existing
OpenClaw or Hermes config.

`type` describes provider identity and product behavior. `api_mode` describes
the wire protocol. Validation must reject incompatible pairs:

| `type` | Allowed `api_mode` values |
| --- | --- |
| `openai` | `openai_chat`, `openai_responses` |
| `anthropic` | `anthropic_messages` |
| `openrouter` | `openai_chat` |
| `gemini` | `google_generate_content` |
| `mistral` | `openai_chat` |
| `custom_openai_compatible` | `openai_chat`, `openai_responses` |

The catalog may omit `default_model` for provider inventory, but v1 runtime
projection for chat agents requires either `default_model` or an explicit
`--default-model` / `--default-provider` render option. Without a concrete
model, `runtime render` fails before materializing runtime files.

The `capabilities` object is declarative metadata used for validation,
projection, and user display. It is not a live provider model catalog. The
user or importing runtime is responsible for its accuracy. Projection adapters
should omit unknown capability-derived fields instead of writing false or zero
sentinels that may change runtime behavior.

Providers may include optional user-owned model hints. This is not a
Clawdi-maintained model catalog. It exists only to preserve metadata imported
from runtimes or entered by the user:

```json
{
  "id": "gpt-5.2",
  "label": "GPT-5.2",
  "input_modalities": ["text", "image"],
  "supports_reasoning": true,
  "context_window": 400000,
  "max_tokens": 128000,
  "cost": {
    "input_per_million": 0,
    "output_per_million": 0
  }
}
```

Projection adapters may use model hints when present and valid. They must omit
unknown model metadata rather than inventing defaults.

## Secret Refs

V1 must support both local env refs and existing Clawdi Vault refs:

```text
env:OPENAI_API_KEY
env:ANTHROPIC_API_KEY
env:OPENROUTER_API_KEY
env:GEMINI_API_KEY
clawdi://default/OPENAI_API_KEY
clawdi://default/openai/api_key
clawdi://project/<project-id>/vault/default/field/OPENAI_API_KEY
clawdi://project/<project-id>/vault/default/section/openai/field/api_key
```

The env var name must match:

```text
^[A-Z][A-Z0-9_]{0,127}$
```

`runtime_env_name` uses the same validation rule.

The `clawdi://` grammar is the existing Vault reference grammar implemented in
`packages/cli/src/lib/secret-references.ts`. Do not redefine it in the AI
Provider layer. For example, `clawdi://default/openai/api_key` means
vault `default`, section `openai`, field `api_key`.

Future resolvers should join the same interface:

```text
keychain:service/account
op://vault/item/field
```

The catalog stores refs, not values. The AI Provider layer should wrap and
reuse the existing secret reference implementation instead of creating a
parallel resolver stack. A resolver can produce either:

1. A runtime-native secret reference, when the engine supports one.
2. An owner-only env file or process env map, when the engine needs env vars.
3. A plaintext value in memory only for direct `test` calls.

Resolved secret values must never be logged, stored in generated config
snapshots, or written into default backups.

Audit language must distinguish secret resolution from model traffic:

1. `env:` refs resolve locally and require no Clawdi API call.
2. `clawdi://` refs resolve through `/api/vault/resolve` and require CLI API
   key auth.
3. After the secret is resolved or injected, BYOK model traffic still goes
   directly from runtime to provider.

## Provider Auth Payloads

Provider auth payloads are part of AI Provider. They may point to an external
ref or to Clawdi-managed encrypted material:

```json
{ "type": "secret_ref", "ref": "env:OPENAI_API_KEY" }
{ "type": "api_key", "source": "vault", "ref": "clawdi://default/openai/api_key" }
{ "type": "oauth_profile", "provider": "codex", "profile": "default" }
{ "type": "agent_profile", "tool": "codex", "profile": "default" }
```

The user-facing object is still the AI Provider. Internally, Clawdi may store
encrypted API key values, OAuth refresh tokens, provider token sets, or whole
tool-specific credential profiles in a separate payload store keyed by
provider ID and auth profile. Runtime adapters decide whether provider auth can
be converted to a process env var, materialized to a tool-owned auth file,
referenced through a runtime-native mechanism, or rejected because the runtime
has no verified way to consume that auth type.

Provider export includes auth metadata and external refs only. Encrypted
secret backup can include provider-auth payloads only when the user explicitly
selects encrypted secret export.

## CLI Commands

First batch:

```bash
clawdi ai-provider list
clawdi ai-provider add
clawdi ai-provider edit <provider-id>
clawdi ai-provider remove <provider-id>
clawdi ai-provider validate [provider-id]
clawdi ai-provider test <provider-id>
clawdi ai-provider connect <provider-id>
clawdi ai-provider complete-oauth <provider-id>
clawdi ai-provider import-auth <provider-id>
clawdi ai-provider materialize-auth <provider-id>
clawdi ai-provider export
clawdi ai-provider import [file]
```

AI Provider command behavior:

- `list` prints provider IDs, type, base URL host, default model, and auth
  availability without printing secret values.
- `add` supports flags for scripted use and prompts when run interactively.
  It may suggest base URLs and env var names by provider type.
- `edit` updates metadata and secret refs, preserving provider ID unless the
  user passes an explicit rename option.
- `remove` refuses removal when a runtime default still points at the provider
  unless `--force` is passed.
- `validate` checks schema, unique IDs, supported provider types, URL shape,
  secret ref syntax, provider auth syntax, default references, and projection
  compatibility. It should not make provider network calls.
- `test` resolves the secret according to ref type and calls the provider
  directly from the CLI. `clawdi://` refs may call Clawdi Vault for secret
  resolution, and managed provider-auth payloads may resolve through the
  internal auth store, but the provider probe itself must target the provider
  base URL, not a Clawdi Cloud model proxy. Provider adapters should use the
  lowest-cost documented probe available and print redacted diagnostics.
- `connect` starts an official OAuth or device-code flow for the provider's
  selected auth type and stores the resulting encrypted payload under that AI
  Provider.
- `import-auth` imports an existing API key, env/vault ref, keychain item, or
  local agent credential profile into the selected AI Provider auth.
- `materialize-auth` writes the selected AI Provider auth into the adapter's
  verified local target, such as Codex `auth.json`, Claude Code credentials, an
  owner-only env file, or a runtime-managed auth store. It should dry-run and
  create backups before replacing existing local files.
- `export` writes metadata and secret refs only by default.
- `import` validates and merges by provider ID. It accepts a provider catalog
  file by default, or a runtime source through `--from-openclaw` /
  `--from-hermes`. Conflicts require explicit replace or rename behavior.

Second batch:

```bash
clawdi runtime render --engine openclaw
clawdi runtime render --engine hermes
clawdi runtime render --engine codex
clawdi runtime inspect
clawdi doctor ai-provider
```

Runtime command behavior:

- `runtime render` reads the catalog, resolves refs as far as the target engine
  needs, and emits deterministic projection files.
- `runtime render --write` materializes deterministic Clawdi-owned projection
  files under the selected output directory or `$CLAWDI_HOME/runtime/<engine>/`.
  It does not patch the user's primary runtime config by default.
- `runtime render --write --activate` asks the target adapter for the least
  invasive activation plan. Activation should prefer a runtime-supported
  config-path env var, include/overlay file, or managed runtime home/profile.
  Native config patching is the last resort and requires the explicit
  `--patch-native` option.
- `runtime inspect` shows catalog source, selected defaults, projected provider
  IDs, missing refs, and target file paths with all secrets redacted.
- `doctor ai-provider` runs catalog validation plus engine projection checks.

## Command Contract Details

`clawdi ai-provider add` should support a fully non-interactive path:

```bash
clawdi ai-provider add <provider-id> \
  --type <type> \
  --base-url <url> \
  --default-model <model> \
  --api-mode <mode> \
  --auth env:<ENV_VAR>|clawdi://...|agent:<tool>/<profile>|oauth:<provider>/<profile>|none \
  --runtime-env <ENV_VAR> \
  --label <label>
```

Interactive `add` asks only for missing fields. It may suggest known base URLs
and env var names, but the written catalog should contain explicit values.
When `--auth` is `env:<ENV_VAR>`, `runtime_env_name` defaults to that env var.
When `--auth` is `clawdi://...`, `--runtime-env` is required unless the target
runtime can consume the verified native secret-ref form directly.
When `--auth` is `agent:<tool>/<profile>` or
`oauth:<provider>/<profile>`, validation checks profile metadata and runtime
projection requires either a materialized env var name or a verified native
runtime auth form.

`clawdi ai-provider list` output columns:

```text
ID              TYPE        DEFAULT MODEL     SECRET REF             STATUS
openai-main     openai      gpt-5.2           env:OPENAI_API_KEY     available
anthropic-work  anthropic   claude-opus-4-6   env:ANTHROPIC_API_KEY  missing
```

`clawdi ai-provider validate` should return non-zero for:

1. Invalid JSON or schema version.
2. Duplicate provider IDs.
3. Invalid provider type or API mode.
4. Incompatible provider type and API mode.
5. Invalid URL shape.
6. Invalid secret ref syntax.
7. Invalid `runtime_env_name`.
8. Defaults pointing to missing providers.
9. Requested engine projection incompatibility when `--engine` is passed.

`auth: none` validation:

1. Allow loopback hosts such as `localhost`, `127.0.0.1`, and `::1` by
   default.
2. Warn for private-network hosts unless the user confirms intent.
3. Fail for public network origins unless the command receives an explicit
   override such as `--allow-no-auth-public`.
4. Projection must not synthesize fake API keys. If a runtime requires a
   placeholder for keyless local servers, the adapter must document that
   runtime-specific marker and keep it out of the catalog.

`clawdi ai-provider test` should accept:

```bash
clawdi ai-provider test <provider-id> [--model <model>] [--timeout <seconds>]
```

The command uses `default_model` unless `--model` is passed. If neither is
available, it performs only a provider-level connectivity probe when a safe
metadata endpoint exists; otherwise it exits with a clear "model required"
error.

Provider tests are adapter-owned and must be conservative:

1. A metadata probe such as a provider model-list endpoint is preferred when it
   can verify auth and endpoint reachability without generating model output.
2. A model-generation probe is allowed only when metadata probing is
   unavailable or the user asks for strict model validation. It must use the
   smallest documented request that exercises the selected protocol.
3. Probe adapters must never send user prompts, files, session data, or runtime
   context. They may send a fixed synthetic prompt such as `ping` only for
   strict generation validation.
4. Redacted diagnostics may include provider host, HTTP status family, protocol,
   model ID, and error class. They must not include request body, response body,
   or secret values by default.
5. Exact probe endpoints and payloads must be checked against official provider
   docs during implementation and pinned in adapter tests.

`clawdi ai-provider import` should accept exactly one source:

```bash
clawdi ai-provider import providers.json
clawdi ai-provider import --from-openclaw <path>
clawdi ai-provider import --from-hermes <path>
```

The default JSON import path expects an AI Provider Catalog or an exported
`ai_provider_catalog` object. Runtime import flags call the matching import
adapter and may produce warnings for inferred protocol, skipped plaintext
secrets, or unsupported runtime fields.

`clawdi runtime render` should accept:

```bash
clawdi runtime render --engine <openclaw|hermes|codex> \
  [--provider <provider-id>] \
  [--default-provider <provider-id>] \
  [--out <dir>] \
  [--write] \
  [--activate] \
  [--patch-native] \
  [--dry-run] \
  [--json]
```

Default behavior writes projection files to stdout or `--out`. `--write`
materializes Clawdi-owned files. `--activate` makes the runtime use those
files through the adapter's safest available activation mechanism.
`--activate` requires `--write`. `--patch-native` is required before an adapter
may modify a primary runtime config file such as `~/.hermes/config.yaml`.

`clawdi runtime inspect` should show:

1. Catalog path and hash.
2. Effective default provider.
3. Provider projection support by engine.
4. Secret ref availability.
5. Generated file targets.
6. Drift between current generated files and the catalog.
7. Whether each provider is direct BYOK or Clawdi-managed.

## `clawdi run` And Runtime Render

`clawdi run` and `clawdi runtime render` solve different parts of the same
problem and should compose.

`clawdi run` is process env injection:

1. It reads env files and inherited env.
2. It resolves `clawdi://` references through the existing Vault resolver.
3. It may resolve provider-auth payloads only for auth kinds that are safe to
   expose as env values, such as API keys or secret-ref wrappers.
4. It injects resolved env values into one child process.
5. It does not permanently change engine-native config or materialize whole
   local auth files.

`runtime render` is config materialization:

1. It reads the Provider Catalog.
2. It writes or prints engine-native provider config as Clawdi-owned
   projection files.
3. It should prefer secret refs or env variable names over plaintext values.
4. It should not launch the engine by itself.
5. It should not mutate the user's primary runtime config unless activation is
   requested and native patching is explicitly allowed.

The two paths compose when projection emits env names. For example, Hermes can
receive `key_env: OPENAI_API_KEY`, while `clawdi run --env-file .env.clawdi --
hermes` can resolve `OPENAI_API_KEY=clawdi://default/openai/api_key` before
launch. If the user starts Hermes without `clawdi run`, the required env var
must already be present through the shell, systemd, Docker, Kubernetes, or the
hosted deployment secret injection layer.

Priority rule:

1. Provider Catalog is the source of truth for provider identity and runtime
   config generation.
2. `clawdi run` is one secret delivery mechanism.
3. Native engine config is generated output unless the user explicitly opts out
   and manages it by hand.

## Provider Auth And Agent Profiles

AI Provider v1 should treat auth as part of the provider. The product goal is
centralized management of provider access, not only provider metadata:

1. API keys entered manually, imported from env/keychain, or stored in Vault.
2. OAuth connections acquired through official provider or tool flows.
3. Local agent credential profiles such as Codex `~/.codex/auth.json`, Claude
   Code credentials, GitHub CLI `hosts.yml`, and runtime-specific auth stores.
4. Lightweight refs such as `env:` and `clawdi://` that point to externally
   managed secret sources.

This means existing `clawdi agent credentials import codex` behavior should
migrate under AI Provider auth, not remain a separate product island. The old
command group should become a compatibility alias around the new provider-auth
service:

```bash
clawdi ai-provider import-auth openai-codex --tool codex --profile default --from ~/.codex/auth.json
clawdi ai-provider materialize-auth openai-codex

# Compatibility aliases:
clawdi agent credentials import codex
clawdi agent credentials materialize codex
```

In the current implementation slice, `clawdi ai-provider import-auth` and
`connect` sync the local provider metadata to `/api/ai-providers`, then store
the encrypted credential payload through
`/api/ai-providers/{provider_id}/auth/import`. `materialize-auth` resolves the
payload through the CLI-only provider-auth resolve route. The older
`clawdi agent credentials` command group still exists as a compatibility path
for existing profile backup/restore behavior, but it is no longer the storage
path for provider-bound credentials.

Suggested provider-auth shape:

```json
{
  "id": "openai-codex",
  "type": "openai",
  "base_url": "https://api.openai.com/v1",
  "api_mode": "openai_responses",
  "auth": {
    "type": "agent_profile",
    "tool": "codex",
    "profile": "default",
    "payload_ref": "ai-provider-auth://openai-codex/default"
  }
}
```

Provider auth types for v1:

```text
secret_ref
api_key
oauth_profile
agent_profile
none
```

`secret_ref` auth values are wrappers around `env:` or `clawdi://` refs. They
are useful when the user wants the same Provider ID while the underlying secret
source varies per environment.

AI Provider owns the auth object:

```json
{
  "id": "openai-main",
  "type": "openai",
  "base_url": "https://api.openai.com/v1",
  "default_model": "gpt-5.2",
  "api_mode": "openai_responses",
  "auth": {
    "type": "api_key",
    "source": "vault",
    "ref": "clawdi://default/openai/api_key"
  }
}
```

Rules:

1. Provider Catalog owns endpoint metadata and runtime capabilities.
2. Provider auth owns refs to encrypted token material and tool-specific auth
   payloads.
3. Runtime adapters decide how provider auth is consumed: env injection,
   runtime-native ref, or explicit materialization to a tool auth file.
4. Whole local auth files must never be exposed through all-env injection or
   `/api/vault/resolve`. They can only be materialized by a user-level command
   or hosted agent runtime path that explicitly requests that provider auth
   type.
5. Environment-bound API keys cannot list or materialize local agent credential
   profiles; user-level auth is required.

### OAuth Connect Flow

OAuth must be implemented by Clawdi backend/provider adapters, not by spawning
official local agent CLIs. The CLI and dashboard both call the same backend
surface:

```bash
clawdi ai-provider connect openai-codex --method oauth --tool codex --profile default
clawdi ai-provider connect anthropic-main --method oauth --profile work
clawdi ai-provider connect claude-code-main --method oauth --tool claude-code --profile default
```

The backend adapter must choose the safest official flow available for that
provider:

1. Authorization-code + PKCE when the provider supports it.
2. Device-code flow when the provider/tool supports device auth.
3. Manual paste completion when loopback is unavailable and the provider
   supports pasting a code or callback URL.

Do not implement OAuth by scraping browser cookies, reading private browser
profiles, reverse-engineering unsupported token stores, or shelling out to
`codex login`, `claude auth login`, or similar agent commands. Before enabling
a provider adapter in production, Phase 0.5 must verify the official current
contract: authorization URL, token exchange, refresh behavior, scopes, token
file shape, and whether a public client or backend confidential client is
required.

Backend OAuth uses short-lived encrypted state:

```text
POST /api/ai-providers/{provider_id}/auth/oauth/start
POST /api/ai-providers/{provider_id}/auth/oauth/complete
```

`start` returns the authorization link directly:

```json
{
  "provider_id": "openai-codex",
  "oauth_provider": "codex",
  "profile": "default",
  "auth_url": "https://...",
  "state": "encrypted-state",
  "redirect_uri": "https://cloud.example/oauth/callback",
  "expires_at": "2026-06-01T12:00:00Z"
}
```

`complete` accepts the encrypted state plus the provider code. The CLI default
flow listens on `127.0.0.1:<ephemeral>/callback`, opens or prints the
`auth_url`, captures the browser callback, and forwards only `code`, `state`,
and `redirect_uri` to the backend. When loopback is unavailable, manual mode
still asks the provider to redirect to a loopback URL such as
`http://127.0.0.1:14565/callback`; the user can paste that full failed redirect
URL into `clawdi ai-provider complete-oauth <provider-id> --redirect-url
<url>`. The backend performs the token exchange and stores the encrypted
provider-auth payload. The CLI should not exchange OAuth tokens locally for
these provider-auth flows.

Security requirements:

1. Bind OAuth `state` and PKCE verifier to the authenticated Clawdi user and a
   short TTL. Do not expose the PKCE verifier in readable URL state; use
   encrypted state or a server-side pending state store.
2. Store refresh tokens and auth files encrypted with the same key-separation
   principles as Vault.
3. List only redacted metadata: provider ID, profile, scopes, expiry, status, and
   last materialized target.
4. Never include OAuth access tokens, refresh tokens, auth files, or API keys
   in default backup, logs, diagnostics, runtime projection snapshots, or
   Cloud payload metadata.
5. Secret backup may include provider-auth payloads only through explicit
   encrypted export.
6. Support revocation and local deletion separately: removing a local
   materialized file should not necessarily revoke the stored provider auth,
   and revoking provider auth should mark the Provider as unavailable.

## Gateway Relationship

Gateway should be modeled as an AI Provider only for Clawdi-managed traffic,
not for user BYOK traffic.

```json
{
  "id": "clawdi-managed",
  "type": "custom_openai_compatible",
  "label": "Clawdi managed",
  "base_url": "https://gateway.example.com/openai/v1",
  "default_model": "gpt-5.2",
  "api_mode": "openai_chat",
  "auth": {
    "type": "secret_ref",
    "ref": "env:CLAWDI_GATEWAY_TOKEN"
  },
  "managed_by": "clawdi"
}
```

Rules:

1. User BYOK providers must not be rewritten to Gateway.
2. Gateway providers are explicit catalog entries with `managed_by: "clawdi"`
   or equivalent metadata.
3. Gateway can preserve zero-config onboarding for users who choose
   Clawdi-managed routing.
4. Search, transcription, images, and other non-chat Gateway services should
   be separate provider capability work; this v1 catalog focuses on model
   providers used by agent runtimes.

## Runtime Flow

Startup flow:

1. CLI reads the Provider Catalog.
2. CLI validates selected providers and defaults.
3. CLI resolves secret refs into runtime-native refs, env maps, or in-memory
   values as required by the target engine.
4. CLI generates OpenClaw or Hermes projection files.
5. If requested, CLI activates those files through the runtime adapter's
   safest supported mechanism.
6. OpenClaw or Hermes reads the generated config and calls provider APIs
   directly.
7. Clawdi Cloud is not in the model request path for BYOK traffic.

## System Architecture

The design has one provider source of truth and multiple generated outputs.

```text
Provider Catalog
  -> catalog validation
  -> provider normalization
  -> secret ref resolution / availability inspection
  -> runtime projection adapter
  -> engine-native generated files
  -> runtime activation adapter
```

Responsibilities:

1. **Provider Catalog** stores user intent: provider IDs, type, endpoint,
   default model, API mode, capabilities, and auth refs.
2. **Catalog validation** checks schema, references, compatibility, and
   projection preconditions.
3. **Provider normalization** converts provider-specific vocabulary into a
   small internal contract that projection adapters can consume.
4. **Secret ref resolution** turns refs into availability status, runtime-native
   refs, env maps, or in-memory values depending on the command.
5. **Runtime projection adapters** generate OpenClaw, Hermes, or future engine
   config from normalized providers without mutating primary runtime config.
6. **Runtime activation adapters** make generated files discoverable through
   runtime-supported config paths, include mechanisms, managed homes/profiles,
   or explicit native patches.
7. **Import adapters** reverse existing runtime config back into catalog
   entries when possible.

Do not write provider-by-runtime adapters. The product needs one projection
adapter per runtime, not one adapter per `(provider, runtime)` pair.

Wrong shape:

```text
OpenAIToHermesAdapter
AnthropicToHermesAdapter
OpenRouterToOpenClawAdapter
GeminiToOpenClawAdapter
```

Correct shape:

```text

HermesProjectionAdapter
OpenClawProjectionAdapter
HermesImportAdapter
OpenClawImportAdapter
```

The normalizer owns provider semantics that are independent of a runtime. The
projection adapter owns only the target runtime's config grammar.

## Provider Normalization

Projection adapters should not branch on every provider's raw catalog shape.
They should consume a normalized shape.

```ts
export interface NormalizedProvider {
  id: string;
  label: string;
  type: AiProviderType;
  baseUrl: string;
  defaultModel?: string;
  protocol: "openai-chat" | "openai-responses" | "anthropic-messages" | "google-generate-content";
  auth: NormalizedProviderAuth;
  managedBy: "user" | "clawdi";
  capabilities: AiProviderCapabilities;
  models: NormalizedProviderModel[];
}

export type NormalizedProviderAuth =
  | {
      mode: "secret-ref";
      ref: ParsedSecretRef;
      preferredEnvName?: string;
    }
  | { mode: "none" };

export interface NormalizedProviderModel {
  id: string;
  label: string;
  protocol?: NormalizedProvider["protocol"];
  inputModalities?: Array<"text" | "image" | "video" | "audio">;
  supportsReasoning?: boolean;
  contextWindow?: number;
  maxTokens?: number;
  cost?: AiProviderModelCost;
  capabilities?: AiProviderCapabilities;
}
```

Normalization rules:

1. `api_mode` determines wire protocol; `type` determines provider identity,
   UX suggestions, and validation constraints.
2. `base_url` is canonicalized by trimming trailing slashes.
3. Provider IDs stay exactly user-owned after validation.
4. Default env var names are suggestions for `add`, not hidden defaults during
   projection.
5. No default model table lives in the normalizer.
6. Provider compatibility quirks that affect request shape belong in the
   protocol mapping, not in Cloud.
7. Capability booleans stay tri-state at the projection boundary: true, false,
   or unknown/omitted. Unknown must not be projected as false.
8. Model hints are normalized only from catalog data or import data. The
   normalizer must not fetch or maintain a provider model catalog.

Protocol mapping from the local runtime contract audit:

| `api_mode` | Normalized protocol | OpenClaw API | Hermes API mode |
| --- | --- | --- | --- |
| `openai_chat` | `openai-chat` | `openai-completions` | `chat_completions` |
| `openai_responses` | `openai-responses` | `openai-responses` | `codex_responses` |
| `anthropic_messages` | `anthropic-messages` | `anthropic-messages` | `anthropic_messages` |
| `google_generate_content` | `google-generate-content` | `google-generative-ai` | fixture required |

Hermes currently documents three generic API modes for custom providers:
`chat_completions`, `codex_responses`, and `anthropic_messages`. A generic
custom-provider `gemini` API mode is not documented in the audited Hermes
version. Gemini projection must therefore use a verified native-provider
fixture or remain unsupported for Hermes until such a fixture exists.

These labels must still be pinned by golden tests against the exact OpenClaw
and Hermes versions Clawdi targets.

## Secret Resolution Contract

Secret handling has three modes:

1. **Inspect mode**: report whether a ref is configured and available. Never
   read plaintext if metadata is enough.
2. **Projection mode**: emit runtime-native refs, env var names, or owner-only
   env files. Do not embed plaintext in generated config when the runtime can
   consume a ref.
3. **Test mode**: resolve plaintext into memory only, call the provider
   directly, then discard it.

Implementation must reuse the existing Clawdi reference resolver:

1. `env:` uses local process env inspection/resolution.
2. `clawdi://` uses `parseClawdiReference`, `previewClawdiReference`, and
   `resolveClawdiReference` from `packages/cli/src/lib/secret-references.ts`.
3. Bulk preview/resolve should use the existing bulk resolver where possible,
   so `ai-provider validate`, `runtime inspect`, `read`, `inject`, and `run`
   report the same provenance and conflict behavior.

Suggested result type:

```ts
export type SecretResolution =
  | {
      status: "available";
      ref: ParsedSecretRef;
      value?: string;
      runtimeRef?: unknown;
    }
  | {
      status: "configured_unavailable";
      ref: ParsedSecretRef;
      reason: string;
    }
  | {
      status: "unsupported";
      ref: ParsedSecretRef;
      reason: string;
    };
```

Security rules:

1. Logs and thrown errors must include refs, not values.
2. Dry-run uses inspect mode by default.
3. `test` is the only v1 command that needs plaintext.
4. Default backup uses refs only.
5. Optional secret backup must encrypt before writing backup artifacts.
6. Privacy claims about "not hitting Clawdi Cloud" apply to model provider
   traffic, not to `clawdi://` secret resolution.

## Adapter Activation Strategy

Adapter implementation should use a three-step model:

1. **Render**: pure transformation from normalized catalog to runtime-native
   files. No disk reads or writes except optional fixture loading in tests.
2. **Materialize**: write generated files that Clawdi owns. The default target
   is `$CLAWDI_HOME/runtime/<engine>/`, but an adapter may choose an
   engine-local managed subdirectory when the runtime's include security model
   requires same-tree files.
3. **Activate**: make the runtime discover the materialized files. Activation
   is explicit and engine-specific.

Activation order:

1. Use a per-process config path env var when the launch path is controlled by
   `clawdi run`, Cloud, Docker, systemd, or a wrapper script.
2. Use the runtime's own non-interactive CLI when it has a stable structured
   patch contract.
3. Use a runtime include/overlay mechanism when the runtime supports one.
4. Use a managed runtime home or profile when that is the runtime's supported
   isolation primitive.
5. Patch the primary native config only as a last resort, only after a dry-run
   diff, only with `--patch-native`, and only through a structured merge of
   verified provider-owned sections.

This keeps the adapter interface deep: callers ask for "make this catalog
usable by engine X," while the adapter hides the runtime-specific activation
mechanism. The test surface is the rendered files plus the activation plan, not
the user's entire home directory.

Runtime CLI activation changes the dependency from a file-format contract to a
command contract. That is preferable when the command contract is strong, but
it is not zero knowledge. The adapter still needs to know which runtime fields
represent providers, defaults, secret refs, and API modes. It just delegates
parsing, validation, migration, locking, and write safety to the runtime.

A runtime CLI activation path is eligible only when the pinned runtime version
provides:

1. Non-interactive execution.
2. Structured input through JSON/stdin/file or batch arguments, not shell
   string interpolation.
3. Dry-run or validation before writes.
4. Additive merge semantics for provider maps.
5. Stable exit codes and diagnostics.
6. A way to target the intended config path, home, or profile.
7. No requirement to pass plaintext API keys.

If a runtime CLI lacks those properties, the adapter may still show the user
manual commands, but automated activation should choose managed files,
profiles, or explicit native patch mode instead.

## Runtime Configuration Mode And Version Pins

Every runtime activation path is a versioned contract. The adapter may render
managed files without native activation, but it may not patch or activate a
runtime through an unpinned setting surface.

| Engine | Best v1 setting mode | Pinned support range | Status | Reason |
| --- | --- | --- | --- | --- |
| Codex | Write `$CODEX_HOME/clawdi-ai-provider.config.toml`; user runs `codex --profile clawdi-ai-provider` | `@openai/codex <1.0.0` when the installed version supports profile config, `model_providers`, and `wire_api = "responses"` | Enabled | Codex profiles are first-class config layers. They avoid editing the user's primary `config.toml`, project config cannot override provider/auth/profile keys, and `-c key=value` is too ephemeral for persistent provider management. |
| Hermes | `hermes config set` under explicit `runtime render --write --activate`; preserve the managed projection and sidecar under `$CLAWDI_HOME/runtime/hermes/` | `Hermes Agent >=0.15.1 <0.16.0` | Enabled for non-dotted provider IDs | The runtime CLI owns native write semantics and has been tested to preserve existing `mcp_servers.clawdi`. Direct YAML replacement is not allowed. |
| OpenClaw | Managed JSON projection only; native activation disabled until config CLI/include fixtures are pinned | Unverified | Render-only | Local source review is promising, but activation depends on OpenClaw's exact schema/CLI contract and must not be guessed. |

Rejected Codex modes:

1. Editing `$CODEX_HOME/config.toml` directly is rejected because that file is
   user-owned and may contain unrelated preferences, auth routing, MCP, or
   model defaults.
2. Project `.codex/config.toml` is rejected for provider management because
   Codex intentionally does not allow project config to override
   provider/auth/profile keys.
3. Passing many `codex -c ...` overrides is rejected for persistent
   activation. It remains useful for one-off debugging, but it is not a stable
   source of truth.

V1 projection support matrix:

| Protocol | OpenClaw | Hermes | Codex |
| --- | --- | --- | --- |
| OpenAI chat-compatible | Supported after fixture lock | Supported after fixture lock | Unsupported unless the endpoint also supports Codex Responses wire API |
| OpenAI Responses-compatible | Supported after fixture lock | Supported after fixture lock as `codex_responses` | Supported through `wire_api = "responses"` |
| Anthropic Messages | Supported after fixture lock | Supported after fixture lock | Unsupported |
| Google Generate Content / Gemini | Supported after fixture lock | Unsupported until a native Gemini fixture is captured | Unsupported |

If a catalog contains a provider that a target runtime cannot represent,
`runtime render --engine <engine>` must fail before writing and name the

## Generated File Ownership

Generated runtime files need explicit ownership markers so future commands can
detect drift without clobbering user-owned edits.

The primary generated location is `$CLAWDI_HOME/runtime/<engine>/`. If a
runtime can include only files under its own config directory, the adapter may
materialize a Clawdi-owned subdirectory there, for example
`~/.openclaw/clawdi/`, but it must still write sidecar metadata and treat the
runtime's primary config file as user-owned.

Each generated file should pair with metadata in a sidecar file:

```json
{
  "managed_by": "clawdi",
  "source": "ai-provider-catalog",
  "catalog_hash": "sha256:...",
  "engine": "hermes",
  "generated_at": "2026-06-01T00:00:00Z"
}
```

Rules:

1. `runtime render` prints files to stdout or a target directory by default.
2. `runtime render --write` writes Clawdi-owned managed files and sidecars. It
   does not patch primary native config.
3. `runtime render --write --activate` may write a small activation file,
   env-file, include target, or managed runtime home depending on adapter
   support.
4. Patching a primary native config file requires `--patch-native`, a redacted
   diff, and a backup unless `--no-backup` is explicitly passed.
5. If a managed file's stored `catalog_hash` differs from the current
   projection, `runtime inspect` reports drift.
6. If a native file has no Clawdi marker, the CLI treats it as user-owned and
   requires explicit confirmation before patching.
7. Do not embed drift markers into engine-native config files unless the
   pinned engine schema explicitly allows unknown metadata keys. Use sidecar
   files by default.

## OpenClaw Projection

Local OpenClaw source review confirms these initial projection facts:

1. Custom model providers live under `models.providers.<provider-id>`.
2. Custom providers must declare `baseUrl` and `models`.
3. `apiKey` accepts OpenClaw `SecretInput`, including
   `{ "source": "env", "provider": "default", "id": "OPENAI_API_KEY" }`.
4. Supported API labels include `openai-completions`, `openai-responses`,
   `anthropic-messages`, and `google-generative-ai`.
5. The input schema allows provider model metadata such as `reasoning`,
   `input`, `cost`, `contextWindow`, and `maxTokens` to be omitted. If present,
   `contextWindow` and `maxTokens` must be positive.
6. `agents.defaults.model` accepts model refs such as
   `openai-main/gpt-5.2`.
7. OpenClaw supports `$include` directives, `OPENCLAW_CONFIG_PATH`, and
   `OPENCLAW_INCLUDE_ROOTS`.

These facts still need Phase 0.5 fixtures against the exact pinned OpenClaw
version before implementation. Projection should preserve env refs instead of
writing plaintext keys when the runtime supports that form.

Expected minimal output:

```json
{
  "models": {
    "providers": {
      "openai-main": {
        "baseUrl": "https://api.openai.com/v1",
        "api": "openai-responses",
        "apiKey": {
          "source": "env",
          "provider": "default",
          "id": "OPENAI_API_KEY"
        },
        "models": [
          {
            "id": "gpt-5.2",
            "name": "gpt-5.2",
            "api": "openai-responses"
          }
        ]
      }
    }
  },
  "agents": {
    "defaults": {
      "model": "openai-main/gpt-5.2"
    }
  }

```

The exact serialized output must be pinned by golden tests against OpenClaw's
current config contract. Placeholder model metadata such as `cost`,
`contextWindow`, and `maxTokens` must be omitted unless the user catalog
contains verified values. Projection must not write zero sentinels.

OpenClaw activation strategy:

1. For hosted or wrapper-controlled launches, prefer a generated managed config
   path and launch with `OPENCLAW_CONFIG_PATH=<managed-config>`. If that config
   includes the user's existing config or other external files, set
   `OPENCLAW_INCLUDE_ROOTS` to the exact allowed directories.
2. For local persistent activation, prefer OpenClaw's own config CLI when the
   pinned version supports the needed shape. The current reviewed surface
   includes `openclaw config patch --stdin --dry-run`, `openclaw config set
   --batch-json`, `--merge`, and protected-map replacement guards.
3. Use argv arrays and stdin/file input when invoking OpenClaw. Do not build a
   shell command string.
4. If the CLI route cannot express an include/overlay install cleanly, use an
   OpenClaw `$include` overlay. The adapter may write Clawdi-owned include
   files and install only the smallest include pointer needed in the primary
   OpenClaw config.
5. If the target OpenClaw section already contains user-authored non-include
   content, activation must not replace it automatically. It should report the
   conflict and require an explicit native patch mode or manual migration.
6. OpenClaw native patch activation is not the default path. It exists only for
   migration edge cases after Phase 0.5 fixtures prove the patch is safe.

## Hermes Projection

Local Hermes source and docs review confirms these initial projection facts:

1. Hermes resolves `config.yaml` from `HERMES_HOME`.
2. Hermes accepts keyed `providers` and normalizes them into the same runtime
   view as legacy `custom_providers`.
3. `custom_providers` remains a legacy on-disk shape that Hermes can read, but
   new generated config should prefer keyed `providers` to avoid duplicated
   provider entries.
4. Hermes accepts `base_url`, `name`, `model` / `default_model`, `models`,
   `key_env`, `api_key_env`, `api_mode`, `extra_body`, and related custom
   provider fields.
5. `api_key_env` is normalized to `key_env`; projection should write
   canonical `key_env`.
6. Generic custom-provider API modes documented in the audited version are
   `chat_completions`, `codex_responses`, and `anthropic_messages`.
7. Unknown keys are ignored with warnings, but projection should not rely on
   unknown-key tolerance for drift markers.

These facts still need Phase 0.5 fixtures against the exact pinned Hermes
version before implementation.

Expected minimal output:

```yaml
model:
  default: "gpt-5.2"
  provider: "openai-main"
  base_url: "https://api.openai.com/v1"

providers:
  openai-main:
    name: "OpenAI main"
    base_url: "https://api.openai.com/v1"
    key_env: "OPENAI_API_KEY"
    model: "gpt-5.2"
    api_mode: "codex_responses"
```

If Hermes requires a provider-specific built-in slug for a native transport,
the projection adapter owns that mapping. Cloud must not duplicate it. If the
target provider requires an env file, the CLI may generate an owner-only env
file and point Hermes at it, but generated files remain disposable runtime
artifacts.

Hermes activation strategy:

1. For hosted or wrapper-controlled launches, prefer a generated Hermes home or
   profile and launch with `HERMES_HOME=<managed-home>` when the runtime should
   be isolated.
2. For local user installs, do not replace `~/.hermes/config.yaml`.
3. The audited `hermes config set` command is useful for scalar values and
   secrets, but it is not enough for automated provider activation by default
   because it lacks a verified structured batch patch and dry-run contract.
4. If Hermes adds a `config patch --stdin --dry-run` or dedicated
   provider-import command, the Hermes adapter should prefer that runtime CLI
   path over native YAML patching.
5. If the pinned Hermes version gains a verified include/profile mechanism that
   can layer provider config over an existing home, use that before native
   patching.
6. If no such mechanism exists, local activation may use native patch mode
   only with `--patch-native`. It must perform structured YAML merge, not
   whole-file replacement. It may update `model` and verified provider-owned
   sections only. It must preserve `mcp_servers`, including the `clawdi` block
   written by `clawdi setup`, and it must use a sidecar drift marker instead
   of adding Clawdi metadata keys to `config.yaml`.

## Codex Projection

Codex support is a runtime projection adapter, not only credential import. It
lets the same Provider Catalog produce a Codex profile that can be selected at
launch time:

```bash
clawdi runtime render --engine codex --write --activate
codex --profile clawdi-ai-provider
```

Verified Codex v1 feature contract for `@openai/codex <1.0.0`:

1. Codex reads user config from `$CODEX_HOME/config.toml`.
2. Codex reads profile files from `$CODEX_HOME/<profile>.config.toml` when the
   user passes `codex --profile <profile>`.
3. Codex supports `model`, `model_provider`, and
   `[model_providers.<id>]` provider definitions.
4. Custom providers require `base_url` and currently use
   `wire_api = "responses"` for the provider wire protocol.
5. Auth can be expressed as `env_key`, or as `requires_openai_auth = true`
   when using Codex native OpenAI login through a compatible endpoint.
6. Project `.codex/config.toml` cannot be used for provider/auth/profile
   management because Codex excludes those keys from project override.

Expected env-backed output:

```toml
# Generated by Clawdi. Do not put API keys in this file.
# Contract: @openai/codex <1.0.0 with profile config, model_providers, and responses wire_api support; $CODEX_HOME/clawdi-ai-provider.config.toml selected with codex --profile.

env_key = "OPENAI_API_KEY"
```

Expected native Codex-auth output for the built-in OpenAI endpoint:

```toml
# Generated by Clawdi. Do not put API keys in this file.
# Contract: @openai/codex <1.0.0 with profile config, model_providers, and responses wire_api support; $CODEX_HOME/clawdi-ai-provider.config.toml selected with codex --profile.
model = "gpt-5.2"
model_provider = "openai"
```

Codex projection rules:

1. Only `api_mode = openai_responses` providers are eligible in v1.
2. OpenAI, OpenRouter, or custom OpenAI-compatible providers that only expose
   Chat Completions are rejected until Codex supports another stable
   `wire_api` value or the provider endpoint is marked Responses-compatible.
3. `auth: agent:codex/<profile>` may use Codex native auth. For the default
   OpenAI base URL, the adapter uses built-in `model_provider = "openai"` and
   does not create a custom provider table.
4. Env, Vault, and managed API key auth project to an env var name through
   `env_key`. The runtime process must receive that env var from the shell,
   hosted secret injection, or `clawdi run`.
5. `oauth_profile` and non-Codex agent profiles are rejected until Codex has a
   verified projection contract for them.

Codex activation strategy:

1. `runtime render --write` materializes
   `$CLAWDI_HOME/runtime/codex/ai-providers.codex.toml` plus sidecar metadata.
2. `runtime render --write --activate` copies the rendered TOML into
   `$CODEX_HOME/clawdi-ai-provider.config.toml` with owner-only permissions.
3. Activation prints the exact launch instruction
   `codex --profile clawdi-ai-provider`.
4. The adapter never edits `$CODEX_HOME/config.toml` and never writes project
   `.codex/config.toml`.

## Backup And Restore

The audited CLI does not currently expose a general `clawdi backup` /
`clawdi restore` command. If that product surface is not introduced in the
same release, `clawdi ai-provider export` and `clawdi ai-provider import`
should serve as the provider-only backup/restore path for v1. The rules below
apply to both a future global backup and provider-only export.

Default backup includes provider metadata, auth refs, and redacted auth
metadata:

```json
{
  "ai_provider_catalog": {
    "schema_version": 1,
    "providers": [
      {
        "id": "openai-main",
        "type": "openai",
        "base_url": "https://api.openai.com/v1",
        "default_model": "gpt-5.2",
        "api_mode": "openai_responses",
        "auth": {
          "type": "api_key",
          "source": "vault",
          "ref": "clawdi://default/openai/api_key"
        }
      }
    ]
  }
}
```

Default backup must not include plaintext secret values, OAuth tokens, refresh
tokens, or whole local auth files.

Optional secret backup requires explicit encryption:

```bash
clawdi backup --include-secrets --secret-passphrase
clawdi backup --include-secrets --secret-recipient age1...
```

Secret export rules:

1. Secrets and provider-auth payloads are resolved only from supported refs or
   provider IDs selected by the user.
2. The secret bundle is encrypted before it enters the backup archive.
3. Metadata can identify which provider/ref/auth profile each encrypted secret
   belongs to.
4. Plaintext never appears in the zip, logs, temp files, or JSON metadata.
5. Restore requires an explicit target such as env file, local keychain, Clawdi
   Vault, AI Provider auth, agent credential profile materialization, or a
   future 1Password adapter.

Restore examples:

```bash
clawdi restore backup.zip
clawdi restore backup.zip --secrets-target env-file --out ~/.clawdi/providers.env
clawdi restore backup.zip --secrets-target vault --project personal
```

Restoring to an env file writes owner-only files and does not edit shell
startup files.

## Backend Storage Model

The local catalog is enough for CLI-only v1, but the hosted agent service needs
a server-side representation from Clawdi Cloud. V1 backend storage should treat
AI Providers as account-global resources: a provider belongs to the user's
account and is available to that user's Projects and hosted agent runtimes by
default. Project-scoped provider ownership, per-Project overrides, and Project
sharing rules can be added later without changing the v1 local catalog format.

The backend should use the same shared schema and store provider metadata
separately from encrypted auth payloads. AI Providers are account-global v1
resources. Project-specific access policy can be added as a grant/override
layer later.

Suggested backend entities:

```text
ai_provider_auth_payloads
  id                    uuid
  owner_user_id          text
  provider_id            text
  scope                  text     # account_global in v1
  auth_profile           text     # default, codex, claude-code, etc.
  kind                   text     # api_key, oauth_profile, agent_profile
  source                 text     # manual, oauth_device_code, oauth_pkce, import_file, keychain, env, vault
  payload_ref            text     # encrypted storage pointer; never plaintext
  metadata               jsonb    # scopes, expiry, profile, target hints, status
  created_at             timestamptz
  updated_at             timestamptz
  archived_at            timestamptz

ai_providers
  id                    uuid
  owner_user_id          text
  scope                  text     # account_global in v1
  provider_id            text     # user-stable ID, e.g. openai-main
  type                   text
  label                  text
  base_url               text
  default_model          text
  api_mode               text
  capabilities           jsonb
  auth_type              text     # secret_ref, api_key, oauth_profile, agent_profile, none
  auth_ref               text     # env:, clawdi://..., or internal payload ref
  auth_metadata          jsonb    # source/profile/tool/scopes; no plaintext secrets
  runtime_env_name       text     # optional injection name, e.g. OPENAI_API_KEY
  created_at             timestamptz
  updated_at             timestamptz
  archived_at            timestamptz
```

The unique key should be `(owner_user_id, provider_id)` for AI Providers in v1.
Auth payloads should be keyed by `(owner_user_id, provider_id, auth_profile)`.
`scope` should be written as `account_global` so a future migration can add
`project_override`, `project_restricted`, or explicit grants without
overloading stable IDs.

Account-global means provider metadata is visible to the account owner's
Projects and hosted agent runtimes by default. It does not mean every Project
owns a copy of the provider, and it does not grant other users access to the
secret material behind provider auth.

Provider auth refs remain separate from provider ownership. A global provider
may reference an env var, a Vault ref, or deployment secret material. If a
Vault ref is Project-exact, the materialization layer must resolve it with
credentials that are allowed to read that Vault. If the ref is relative, the
CLI resolves it using the existing default Project / explicit `--project` /
hosted agent runtime context rules from `secret-references.ts`.

Secret values belong in provider-auth payload storage, Vault, or encrypted
deployment secret storage, not in `ai_providers`. If a Clawdi Cloud onboarding
flow accepts a plaintext API key, it should:

1. Encrypt/store the secret through the provider-auth payload or Vault storage
   path.
2. Create or update an AI Provider whose `auth` object points at that internal
   payload, or at an existing `clawdi://...` Vault ref when the user
   intentionally chooses Vault refs as the source of truth.
4. Store `runtime_env_name` only as the env var name the deployment layer should
   materialize, for example `OPENAI_API_KEY`.
5. Return redacted metadata only.

Do not use `auth_ref = env:OPENAI_API_KEY` as a Cloud storage pointer. `env:`
is a runtime contract; provider-auth payload refs or `clawdi://` refs are the
Clawdi storage references.

Canonical API surface for Cloud/dashboard should be account-scoped:

```text
GET    /api/ai-providers
POST   /api/ai-providers
GET    /api/ai-providers/{provider_id}
PATCH  /api/ai-providers/{provider_id}
DELETE /api/ai-providers/{provider_id}
POST   /api/ai-providers/{provider_id}/auth/api-key
POST   /api/ai-providers/{provider_id}/auth/import
POST   /api/ai-providers/{provider_id}/auth/oauth/start
POST   /api/ai-providers/{provider_id}/auth/oauth/complete
POST   /api/ai-providers/{provider_id}/auth/resolve
POST   /api/ai-providers/{provider_id}/validate
```

Existing `/api/vault/credential-profiles` routes can remain during migration,
but new behavior should route through AI Provider auth. The compatibility layer
maps the old `(tool, profile)` identity to provider auth metadata and preserves
the existing user-level auth requirement for materializing whole local auth
files.

Project-scoped routes should not be part of v1. If a future product surface
needs Project-specific policy, add it as an override layer on top of the
account-global catalog rather than moving the primary provider entity.

`test` needs careful handling. For local BYOK, prefer CLI-side direct test. For
Clawdi Cloud-managed provider secrets, the backend may run a direct provider
probe from server-side infrastructure, but it still must call the provider base
URL directly and must not proxy runtime model traffic.

Deployment secret storage may be implemented outside this OSS repository. The
OSS backend contract should describe provider metadata, Vault refs, and runtime
env names; hosted deployment-specific secret injection can remain an external
caller concern.

## Cloud And Hosted Agent Contract

Clawdi Cloud acts as the OSS control plane. The hosted agent service acts as a
caller/runtime consumer:

1. Onboarding collects provider metadata and secret input.
2. Clawdi Cloud returns OAuth authorization links and completes token exchange
   through `/auth/oauth/start` and `/auth/oauth/complete`; CLI loopback only
   captures and forwards authorization callbacks.
3. Clawdi Cloud stores API keys, OAuth connections, or agent credential
   profiles as encrypted provider-auth payloads.
4. Clawdi Cloud stores provider metadata as an account-global AI Provider with
   auth metadata that references those payloads.
5. Clawdi Cloud encrypts stored secrets according to the existing
   Vault/security model.
6. The hosted agent service injects secrets as env vars, materializes agent
   credential profiles, or uses another resolver-backed secret source.
7. Clawdi Cloud sends the account Provider Catalog and auth metadata/spec to
   the hosted agent service.
8. The hosted agent service invokes the Clawdi CLI to materialize engine
   config.
9. The runtime calls OpenAI, Anthropic, OpenRouter, Gemini, Mistral, or custom
   endpoints directly.

Clawdi Cloud and the hosted agent service should not maintain:

1. OpenClaw provider config format.
2. Hermes provider config format.
3. Provider default model tables.
4. Provider-specific header quirks.
5. Provider-specific API-mode projection logic.

Clawdi Cloud may display provider metadata and validation status, but the OSS
CLI owns projection behavior.

Hosted agent materialization contract:

```json
{
  "ai_provider_catalog": {
    "schema_version": 1,
    "providers": [
      {
        "id": "openai-main",
        "type": "openai",
        "base_url": "https://api.openai.com/v1",
        "default_model": "gpt-5.2",
        "api_mode": "openai_responses",
        "auth": {
          "type": "api_key",
          "source": "vault",
          "ref": "clawdi://default/openai/api_key"
        },
        "runtime_env_name": "OPENAI_API_KEY"
      }
    ]
  },
  "secret_env": {
    "OPENAI_API_KEY": {
      "source": "vault_or_deployment_secret",
      "value_ref": "redacted"
    }
  },
  "runtime": {
    "engine": "hermes",
    "render_command": [
      "clawdi",
      "runtime",
      "render",
      "--engine",
      "hermes",
      "--write",
      "--activate"
    ]
  }
}
```

The runtime receives actual env vars through the deployment secret mechanism,
not through JSON payload fields. The JSON contract above is descriptive of
which env vars must exist; it is not a secret transport.

Cloud UI states:

1. **Not configured**: no providers exist.
2. **Configured, missing auth**: provider metadata exists but its auth ref or
   payload is missing, unavailable, expired, or not materializable.
3. **Configured, missing secret injection**: provider and auth metadata exist
   but hosted agent runtime secret injection is incomplete.
4. **Ready**: provider metadata, credential availability, and secret injection
   are present.
5. **Validation failed**: schema, credential validation, or direct provider
   probe failed.
6. **Runtime projection failed**: CLI materialization reported an engine
   compatibility problem.

## Import And Migration

Migration should be adapter-based and conservative:

1. Read existing OpenClaw config and infer providers from `models.providers`.
2. Read existing Hermes config and infer providers from `providers`,
   `custom_providers`, and `model`.
3. Convert env markers such as `${OPENAI_API_KEY}`, SecretRef objects, and
   Hermes `key_env` into catalog `auth.ref` values.
4. Preserve user-visible provider names as `label`.
5. Generate stable IDs from existing provider keys when possible.
6. If two imported providers collide by ID, prompt or require
   `--rename <old>=<new>`.
7. Never import plaintext `api_key` fields without explicit confirmation and a
   selected secret target.

Import commands:

```bash
clawdi ai-provider import --from-openclaw <path>
clawdi ai-provider import --from-hermes <path>
clawdi ai-provider import-auth openai-codex --tool codex --profile default
clawdi ai-provider import-auth claude-code-main --tool claude-code --profile default
```

Provider import and auth import are separate commands under the same
`ai-provider` product surface. Runtime provider import infers endpoint metadata
and refs. Auth import stores or references the token material needed by those
refs, using explicit user-selected sources and adapter-owned default paths.

## Runtime Integration Boundaries

OpenClaw and Hermes should not need to know about Clawdi Cloud. They only need
engine-native config plus runtime env vars.

For hosted agent runtimes:

1. Clawdi Cloud prepares provider metadata, provider-auth metadata, and secret
   env or profile materialization inputs.
2. The runtime image includes the Clawdi CLI.
3. Entrypoint or setup step writes the Provider Catalog to `$CLAWDI_HOME`.
4. Entrypoint runs
   `clawdi runtime render --engine <engine> --write --activate`.
5. The engine starts normally.

For local runtimes:

1. User manages env vars, Vault refs, provider auth, or future keychain refs
   locally.
2. CLI reads the local Provider Catalog.
3. CLI writes or prints generated config.
4. If the user wants persistent runtime pickup, CLI activation uses the
   adapter's safest mechanism and requires `--patch-native` before editing
   primary native config.
5. The engine starts normally.

For self-hosted runtimes:

1. Operator can mount provider metadata files.
2. Operator can provide env vars or provider-auth payloads through
   Docker/Kubernetes/secret manager.
3. CLI materialization runs at build time, deploy time, or container startup.

No runtime should call Clawdi Cloud for every model request just because the
provider came from a Clawdi catalog.

## Tests

Golden tests are mandatory because projection output is the product contract.

Required coverage:

1. Catalog schema validation accepts valid examples and rejects invalid IDs,
   URLs, invalid `(type, api_mode)` pairs, duplicate IDs, and unsupported
   secret refs.
2. Catalog schema validation accepts optional user-owned `models[]` hints and
   rejects invalid model metadata such as negative context windows or malformed
   cost fields.
3. Secret ref handling supports `env:` refs and delegates `clawdi://` refs to
   the existing Vault reference parser/resolver.
4. Provider-auth handling rejects missing or unsupported auth kinds and never
   exposes whole auth-file payloads through env injection.
5. Pinned OpenClaw provider config fixtures are captured before projection
   tests are written.
6. Pinned Hermes provider config fixtures are captured before projection tests
   are written.
7. OpenClaw projection produces stable config for each provider type after the
   real contract is verified.
8. Hermes projection produces stable config for each provider type after the
   real contract is verified.
9. OpenClaw projection omits unknown model metadata and never writes
   `cost: 0`, `contextWindow: 0`, or `maxTokens: 0`.
10. Hermes projection uses verified API-mode labels such as
   `chat_completions`, `codex_responses`, and `anthropic_messages`.
11. `runtime render --write` materializes Clawdi-owned files and does not
    patch primary native config by default.
12. OpenClaw activation uses verified `OPENCLAW_CONFIG_PATH` or `$include`
    behavior before native patching is allowed.
13. Hermes explicit activation uses the verified Hermes CLI or a structured
    merge and preserves `mcp_servers.clawdi`.
14. Codex projection writes a profile TOML under
    `$CODEX_HOME/clawdi-ai-provider.config.toml`, never edits
    `$CODEX_HOME/config.toml`, includes the pinned `@openai/codex` contract,
    and requires launch through `codex --profile clawdi-ai-provider`.
15. Codex projection rejects non-Responses-compatible providers and projects
    `agent:codex/<profile>` to Codex native auth without leaking auth payloads.
16. Drift metadata is written to sidecar files, not into native config, unless
    the engine schema explicitly permits unknown keys.
17. `export` omits plaintext secrets, OAuth tokens, and whole auth files.
18. Default backup includes provider metadata and redacted auth metadata only.
19. Encrypted secret backup refuses to run without passphrase or public key.
20. OAuth connect tests cover device-code, PKCE/manual paste, expiry, state
    mismatch, and redacted diagnostics with provider-specific fixtures.
21. Agent credential profile compatibility tests prove old
    `clawdi agent credentials` commands route through provider-auth migration
    paths and preserve materialization behavior.
22. Import from OpenClaw and Hermes round-trips representative configs into the
    catalog and back to stable projections.
23. `test` sends provider probes to direct provider URLs for BYOK providers.
    If the auth ref is `clawdi://`, the test may call Clawdi Vault for secret
    resolution before making the direct provider probe. If auth uses an
    encrypted provider-auth payload, the test may resolve that payload before
    making the direct provider probe.
24. `auth: none` validation allows loopback, warns on private networks, and
    blocks public no-auth endpoints without explicit override.

## Backwards Compatibility

Existing users may have:

1. Hermes `model`, `providers`, or `custom_providers` config.
2. OpenClaw `models.providers` config.
3. Env vars referenced directly by runtime config.
4. Plaintext API keys in native config.
5. Existing Clawdi credential profiles stored through
   `/api/vault/credential-profiles`.
6. Local Codex, Claude Code, GitHub CLI, or Hermes auth files that predate the
   provider-auth model.

Compatibility rules:

1. Existing native config keeps working until the user opts into
   `ai-provider import` or `runtime render --write --activate`.
2. Import is additive by default.
3. Runtime render does not overwrite unmarked user config without explicit
   confirmation.
4. Plaintext keys are never silently copied into the Provider Catalog.
5. The CLI should show the exact native provider entries it imported, skipped,
   or could not infer.
6. Existing `clawdi agent credentials` commands continue to work while routing
   through provider-auth migration paths internally.
7. Existing backend credential profiles migrate lazily on first read/write or
   through an explicit migration command; migration must preserve user-level
   auth requirements and materialization behavior.

## Implementation Decisions

These decisions make v1 implementable without waiting for another product
round:

1. Local CLI v1 stores one machine-level catalog under `$CLAWDI_HOME`.
   Cloud v1 stores AI Providers as account-global backend resources and may
   keep encrypted auth payloads in internal tables. Project-specific provider
   overrides and auth grants are deferred.
2. CLI code should reuse the existing Clawdi home helper from
   `packages/cli/src/lib/config.ts`. If needed, expose a public helper instead
   of reimplementing `$CLAWDI_HOME` lookup in the AI Provider module.
3. `ai-provider export/import` is the v1 provider-only backup path unless a
   general `clawdi backup` / `clawdi restore` surface ships first.
4. Encrypted secret backup v1 must support passphrase encryption. Public-key
   recipients are additive and can ship in the same phase if the chosen
   encryption library supports them cleanly.
5. Runtime adapter implementation uses render/materialize/activate as separate
   steps. `--write` materializes managed files; `--activate` connects those
   files to the runtime through a verified runtime CLI or layering mechanism.
   Native config patching remains a separate future mode and must not be
   implied by `--write`.
6. Runtime activation should prefer managed config paths, eligible runtime CLI
   commands, include files, or managed homes/profiles. If the runtime has no
   verified CLI or layering mechanism, activation stays disabled until native
   patching can use a structured merge plus a sidecar marker. Hermes v1
   activation may use `hermes config set`; OpenClaw activation stays gated
   until the pinned CLI or include contract is captured.
7. Model metadata is user-owned optional `models[]` data. Clawdi may validate
   shape and preserve/import/export it, but must not maintain a default model
   metadata table.
8. Existing local agent credential profile commands become compatibility
   aliases around AI Provider auth import/materialization. Codex, Claude Code,
   and GitHub CLI profiles migrate under provider auth rather than becoming a
   separate user-facing catalog.
9. OAuth adapters for Codex, Claude, or provider accounts require a verified
   official contract. Clawdi backend should return the auth link and perform
   token exchange; local agent CLI login commands are not the product path.
10. Gateway search, transcription, image, and other non-chat services stay out
   of AI Provider v1. They can become separate catalog capability work later.

   The backend start/complete shape exists and returns auth links; each
   provider still needs verified client configuration before being enabled.
3. Global `clawdi backup` integration. `ai-provider export/import` is the v1
   provider-only backup surface.
4. OpenClaw native activation.
5. Public-key encrypted secret backup and non-env secret restore targets.

## Implementation Review

Current best-practice assessment after implementation review:

1. **Catalog as source of truth vs per-agent config**: the catalog approach is
   still the right seam. It gives OpenClaw, Hermes, CLI, Cloud, and future
   runtimes one stable provider interface while keeping engine-native files as
   generated outputs.
2. **Provider auth inside AI Provider vs separate credential product**:
   provider auth stays inside AI Provider. Backend payload tables are an
   internal implementation detail, not a second product object. This keeps the
   user model simple while still allowing encrypted API keys, OAuth payloads,
   and whole agent login profiles.
3. **Account-global vs Project-scoped ownership**: account-global remains the
   v1 default. It matches the user's expectation that providers are personal
   account resources available to local runtimes and hosted agent service
   runtimes. Project-scoped policy should be an override layer later, not the
   primary v1 entity.
4. **Render/write/activate split vs direct config mutation**: the split is the
   right runtime interface. `render` is pure, `--write` materializes
   Clawdi-owned files, and `--activate` is only allowed through a verified
   runtime adapter. This avoids hidden edits to user-owned runtime config.
5. **Codex activation through profile file**: the profile file is the best v1
   mode for Codex. It is persistent, launch-selectable, and testable while
   avoiding edits to the user's primary `config.toml`. Project config and
   `-c` overrides are intentionally not used for provider management.
6. **Hermes activation through CLI vs YAML merge**: `hermes config set` is the
   safest v1 activation path because it delegates native write semantics to
   Hermes. The adapter intentionally refuses dotted provider IDs until Hermes
   dot-path escaping is verified.
7. **OpenClaw activation gate vs guessed native contract**: keeping activation
   disabled is correct. The projection file can exist, but native activation
   must wait for pinned fixtures or a verified CLI/include contract.
8. **Secret refs and backup**: metadata-only export by default and
   passphrase-encrypted env-secret export are the right first slice.
   Provider-only backup validates restored metadata before writing env files
   and validates decrypted env names before materialization.
9. **Hermes import parser**: structured YAML parsing is now used for
   `providers` and `custom_providers`; ad hoc line parsing was too brittle for
   real user config.
10. **Public no-auth providers**: loopback no-auth is supported for local
   endpoints, private-network no-auth remains a warning, and public no-auth is
   rejected by writes, imports, tests, and runtime projections.
11. **OAuth acquisition**: backend-owned start/complete is the correct product
    path. The CLI should get an `auth_url` from the backend and never call
    local agent login commands for provider auth. Production provider adapters
    remain gated until official flow contracts and OAuth client settings are
    verified.

User scenario coverage in this slice:

1. **New env-backed OpenAI/OpenRouter/etc. provider**: covered by CLI add/list,
   direct metadata probe, Hermes render, Codex Responses-profile render, and
   e2e secret-leak checks.
2. **Local no-auth model endpoint**: covered for loopback
   `custom_openai_compatible` providers; public no-auth is rejected.
3. **Provider migration from Hermes config**: covered for structured
   `providers`, legacy `custom_providers`, `key_env`, `api_key_env`, labels,
   default provider, and model fields.
4. **Provider migration from OpenClaw projection-shaped JSON**: implemented as
   best-effort import, but native OpenClaw activation remains gated.
5. **Codex/Claude/GitHub-style agent auth profiles**: covered through existing
   credential-profile import/materialization plus AI Provider binding commands.
6. **Encrypted provider-only backup/restore**: covered for env-backed secrets,
   redacted backup output, and owner-only env-file restore.
7. **Cloud-managed API key storage**: covered by backend encrypted
   provider-auth payload routes, redacted dashboard responses, and CLI-only
   plaintext resolve.
8. **Runtime native config preservation**: covered for Hermes by testing that
   activation does not mutate an existing `~/.hermes/config.yaml` containing
   `mcp_servers.clawdi`.
9. **Malformed catalog/import data**: shared validation now reports errors for
   malformed provider entries instead of throwing, and runtime writes reject
   insecure or invalid catalog state.
10. **Unsupported or unverified native activation**: covered by tests proving
    OpenClaw activation and dotted Hermes activation fail before writing managed
    files or invoking runtime CLIs.
11. **Remote invocation readiness**: current command handlers are still a
    Commander-facing adapter. They mix catalog I/O, stdout formatting,
    confirmation prompts, backend calls, and external runtime CLI execution.
    Before adding an RPC surface, the behavior should move behind deeper
    modules with structured inputs/outputs so Commander and RPC become two
    adapters over the same implementation.

## CLI Remote Control And Daemon RPC

Clawdi has three different "remote call" contexts, and they should not share
one vague mechanism:

1. **Cloud metadata calls**: dashboard and backend read/write account-global AI
   Providers through `/api/ai-providers`. This is the right path for provider
   metadata and encrypted payload custody.
2. **Runtime-local materialization**: a local laptop, self-hosted machine, or
   hosted agent runtime needs to render files, invoke runtime CLIs, inspect
   env refs, or materialize auth files. This must execute where the runtime
   filesystem and secret delivery context live.
3. **Agent tool calls**: `clawdi mcp` exposes agent-facing tools over MCP. MCP
   is not the product control plane for mutating runtime/provider config.

V1 should keep remote execution simple: Cloud or hosted deployment code can
invoke the installed CLI binary inside the target runtime environment for
one-shot operations such as:

```bash
clawdi ai-provider import /mnt/config/provider-catalog.json --replace --json
clawdi runtime render --engine hermes --write --activate --json
clawdi runtime inspect --json
```

That is sufficient for deployment-time materialization and avoids adding a new
daemon security surface before the Module boundaries are ready.

A daemon RPC surface is still the right direction for later local/hosted
control-plane operations, but it should be a narrow **CLI Control RPC**, not
"remote shell over CLI":

1. It must expose allowlisted methods, not arbitrary Commander command strings.
2. It must return typed JSON results, warnings, diffs, and redacted diagnostics.
3. It must never return plaintext provider secrets except to a method whose
   explicit purpose is local materialization or direct provider probing inside
   the runtime trust boundary.
4. It must distinguish secret resolution calls from model traffic. RPC may ask
   the CLI to resolve a `clawdi://` ref or managed provider payload, but model
   calls still go runtime to provider.
5. It must be local-first: Unix domain socket on macOS/Linux and named pipe on
   Windows by default. Loopback TCP should require an explicit flag such as
   `--listen 127.0.0.1:<port>` plus a bearer token stored under
   `$CLAWDI_HOME/daemon/control-token`.
6. It must never bind to public interfaces by default. If a hosted agent
   runtime exposes a control channel, Clawdi Cloud should reach it through the
   hosting control plane or a sidecar-local socket, not through an
   internet-facing listener.
7. Mutating methods need idempotency keys, dry-run support, and redacted
   before/after plans. Runtime activation remains separate from render.
8. Interactive OAuth/device-code flows should be modeled as long-running RPC
   jobs with states such as `pending_user`, `completed`, `expired`, and
   `failed`; the RPC should not block an HTTP request for the full user flow.

Suggested first RPC method set:

```text
aiProvider.list
aiProvider.get
aiProvider.upsert
aiProvider.remove
aiProvider.validate
aiProvider.test
runtime.render
runtime.inspect
runtime.activate
providerAuth.importProfile
providerAuth.materialize
doctor.aiProvider
```

Methods such as `providerAuth.resolvePlaintext` should not exist as generic
RPC calls. Plaintext can be used internally by `aiProvider.test`,
`runtime.activate`, or `providerAuth.materialize` after authorization and
redaction rules are applied.

Implementation shape:

1. Extract a deep `ai-provider-service` module for catalog load/save,
   validation, imports, export, direct test planning, and redacted results.
2. Extract a `runtime-materializer` module for projection, write, inspect,
   activation planning, and runtime CLI execution.
3. Extract a `provider-auth-service` module for env/Vault/managed payload
   resolution, profile import/materialization, and OAuth job state.
4. Keep Commander as a thin adapter that parses flags, calls those modules,
   handles prompts, and formats terminal output.
5. Add a separate `daemon control` adapter that speaks JSON-RPC over the local
   control transport and calls the same modules. Do not route RPC through
   Commander or shell command parsing.
6. Keep the existing sync daemon loop focused on sync/SSE/heartbeats. The
   control RPC can live in the same installed daemon process later, but it
   should be implemented as a separate Module with its own auth token, socket,
   logs, and tests. If that makes lifecycle coupling messy, ship it as
   `clawdi control run` first and have `clawdi daemon install` opt into it.

Security tests required before enabling RPC:

1. Socket/token permissions prevent another local user from invoking mutating
   methods.
2. Loopback mode rejects missing/invalid bearer tokens and sends no CORS
   wildcard headers.
3. RPC responses redact env values, Vault values, managed API keys, OAuth
   tokens, and imported credential file contents.
4. Mutating methods can be dry-run without writing files or invoking runtime
   CLIs.
5. `runtime.activate` cannot patch primary runtime config without the same
   explicit activation/patched-native controls as the CLI command.
6. A failed or cancelled long-running OAuth/materialization job leaves no
   partial credential files unless a documented backup/rollback path exists.
