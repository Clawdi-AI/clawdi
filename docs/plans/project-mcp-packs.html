Project MCP Packs

A first-class Clawdi resource for sharing MCP configuration through Projects, resolving credentials from Project Vault, and exposing one combined clawdi mcp server to local agents.

Status: plan Updated: 2026-05-19 Owner: product + platform MVP: local aggregator
Summary User flow Architecture Config Security Data API Roadmap

Summary

Define once

MCP server definitions and Packs are top-level resources for catalog, custom servers, and future provider catalogs.

Install into Projects

Project MCP installations bind definitions to one Project with aliases, overrides, tool policy, and credential bindings.

Run through Clawdi

clawdi mcp fetches runtime config, resolves Vault bindings, starts/connects upstream MCP servers, and exposes namespaced tools.

User flow

Owner configures the shared tool setup. Member attaches the Project to an Agent. The agent sees one Clawdi MCP server.

Owner
1. Choose ProjectStart from Project MCP tab, top-level MCP page, or CLI.
2. Select sourceCatalog, Pack, custom server, or imported .mcp.json.
3. Bind VaultMap secret env/header/arg/url values to Project Vault items.
4. InstallDiscover tools, disable risky tools, save Project installation.
Member
1. Accept shareProject appears in the member inventory.
2. Inspect MCPView metadata, health, enabled tools, and redacted Vault refs.
3. Attach ProjectAdd shared Project as Agent context.
4. Use toolsAgent calls namespaced tools through local clawdi mcp.

Runtime architecture

Ship local aggregation first. Keep the existing Composio Tool Router bridge as a dedicated connector path, not as the generic third-party MCP proxy.

Cloud metadataMCP servers, Packs, Project installations, tool policies.
Project VaultEncrypted credential values and structured bindings.
Runtime config/api/mcp/runtime-config?agent_id=..., CLI/API-key only.
Local clawdi mcpStarts stdio servers, connects remote servers, registers tools, and forwards connector tools.
Composio bridge/api/mcp/composio forwards to user-scoped Tool Router sessions.
Agent clientClaude Code, Codex, Hermes, OpenClaw.

Existing fit

  • Projects and memberships already define sharing.
  • Agent Project bindings already define runtime Project order.
  • Vault plaintext resolution already requires CLI/API-key auth.
  • Connector MCP now uses a backend Composio bridge and preserves upstream tool names and inputSchema.

Security boundaries

Required invariants

  • Dashboard shows metadata and redacted Vault refs only.
  • Plaintext credentials only leave through CLI/API-key runtime endpoints.
  • Agent-bound keys can render only their own Agent and attached Projects.
  • Viewer members can use shared credentials at runtime but cannot edit in v1.
  • Composio API keys and Tool Router session headers never leave the backend bridge.

Explicit risk

  • Local stdio MCP servers are executable code.
  • Injected secrets can be exfiltrated by a malicious upstream server.
  • Do not support shared shell-based credential helpers.
  • Generic hosted proxy still requires separate OAuth, sandbox, revocation, and audit work.
  • Composio bridge tokens are user-scoped connector access, not Project-share authorization.

Data model

mcp_servers

Top-level definitions: source, transport, runtime mode, default command/url, auth config, capabilities, discovery cache.

mcp_packs

Reusable bundles of server definitions with aliases, tool prefixes, enabled defaults, and version pins.

project_mcp_installations

Project usage layer with aliases, overrides, env/header templates, timeouts, and pack provenance.

project_mcp_credential_bindings

Structured mapping from injection targets to value sources: Vault, local env, input, connector. No plaintext literal values.

project_mcp_tool_policies

Per-tool enabled state, risk label, and approval policy.

mcp_runtime_events

Post-MVP audit surface for project, installation, tool, status, and latency.
