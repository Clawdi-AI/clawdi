> *old* transport, which engaged interception via a **forward HTTP proxy plus

Scope: establish whether Clawdi's channel transport is intended to be a transparent outbound egress relay, and identify which explicit channel URL shims are necessary, redundant, or wrong. This document is architecture-only and cites committed/source-controlled code with `file:line` evidence.

## 1. Egress Mechanics

### 1.1 Profile model and route semantics

The Clawdi egress profile schema is explicit: a profile has one of five kinds, `http`, `websocket`, `provider`, `passthrough`, or `deny`; matching can constrain scheme, host, path, headers, and query; rewriting can set an upstream base URL and headers, including sidecar-resolved secret references (`packages/cli/src/runtime/egress-profiles.ts:128`, `packages/cli/src/runtime/egress-profiles.ts:143`, `packages/cli/src/runtime/egress-profiles.ts:169`). The schema requires `http`, `websocket`, and `provider` profiles to have `rewrite.upstreamBaseUrl`, while `deny` and `passthrough` profiles must not carry rewrite rules (`packages/cli/src/runtime/egress-profiles.ts:182`). The generated bundle is `clawdi.egressProfiles.v1`, and Clawdi writes it to the runtime paths only when enabled profiles exist (`packages/cli/src/runtime/egress-profiles.ts:215`, `packages/cli/src/runtime/manifest.ts:3078`).

At request time, the native sidecar matches only enabled profiles loaded from that bundle (`packages/cli/native/runtime-sidecar/main.go:198`). A non-matching request is rejected, not passed through by default (`packages/cli/native/runtime-sidecar/main.go:688`). `deny` returns a blocked response; other matched profiles route to their configured upstream, preserve or rewrite the path, set `Host`, add `X-Clawdi-Original-Host` for non-passthrough traffic, and resolve any rewrite secrets inside the sidecar process (`packages/cli/native/runtime-sidecar/main.go:693`, `packages/cli/native/runtime-sidecar/main.go:697`, `packages/cli/native/runtime-sidecar/main.go:703`, `packages/cli/native/runtime-sidecar/main.go:719`, `packages/cli/native/runtime-sidecar/main.go:722`).

### 1.2 Accepted DNS limitation

The Invisible Gateway allows DNS from the agent UID to the pod resolver so
ordinary provider and channel hostnames continue to resolve before HTTP/TLS
interception. DNS payloads are not inspected by the sidecar, billed, or matched
against egress profiles. An agent can therefore exfiltrate data through DNS query
names, for example by encoding data into labels under an attacker-controlled
domain.

This is an accepted limitation for the controlled/semi-trusted beta launch. It
is not acceptable for untrusted public signup. The fast-follow hardening item is
to replace open resolver access with a DNS allowlist resolver or DNS egress proxy
that permits only approved provider/control-plane/channel domains and blocks or
logs all other query names.

### 1.3 Engagement mechanism: proxy environment plus CA trust

egress engagement is not iptables/NAT interception in the code reviewed. Clawdi engages interception by starting a local HTTP forward proxy and injecting proxy and CA environment variables into the runtime process. The runtime-facing environment includes `HTTPS_PROXY`, `HTTP_PROXY`, lowercase proxy variants, `NO_PROXY`, `OPENCLAW_PROXY_URL`, and CA variables for Node, Python requests, curl, git, Deno, and Codex (`packages/cli/src/runtime/egress-env.ts:58`, `packages/cli/src/runtime/egress-env.ts:64`, `packages/cli/src/runtime/egress-env.ts:65`, `packages/cli/src/runtime/egress-env.ts:66`). Hosted systemd runtime programs receive the same runtime env from `applyMitmSidecarRuntimeEnv` when a egress program exists (`packages/cli/src/runtime/manifest.ts:2254`). `clawdi run -- ...` uses the same mechanism: it builds an invocation env from the hosted run config, starts the sidecar if `CLAWDI_EGRESS_ENABLED=1`, applies the sidecar's actual proxy URL and CA, strips sidecar control env, and spawns the child with that environment (`packages/cli/src/runtime/run-config.ts:244`, `packages/cli/src/commands/run.ts:347`, `packages/cli/src/commands/run.ts:367`, `packages/cli/src/commands/run.ts:380`).

Therefore a process is intercepted only if it sends traffic through the injected HTTP proxy environment, or through runtime-specific code that consumes `OPENCLAW_PROXY_URL`, and trusts the generated CA for TLS. A process or library that ignores `HTTPS_PROXY`/`HTTP_PROXY` for WebSocket or TLS sockets will not be transparently intercepted by this layer even if it inherits the environment. Conversely, a process that honors those env vars does not need explicit Clawdi relay URLs for the same destination.

### 1.4 Sidecar process and native implementation

The published CLI package carries a bundled `clawdi-runtime-sidecar` artifact; `prepublishOnly` builds it from `packages/cli/native/runtime-sidecar` into `clawdi-runtime-sidecar`, and that directory is included in the package files (`packages/cli/package.json:11`, `packages/cli/package.json:36`, `packages/cli/scripts/build-runtime-sidecar-bundle.mjs:9`, `packages/cli/scripts/build-runtime-sidecar-bundle.mjs:31`, `packages/cli/scripts/build-runtime-sidecar-bundle.mjs:42`). I found no separate source path or package named `clawdi-egress-broker`; in this repo the broker role is the native sidecar binary.

The TypeScript launcher resolves either `CLAWDI_EGRESS_SIDECAR_PATH`, `CLAWDI_EGRESS_SIDECAR_BUNDLE/bin/clawdi-runtime-sidecar`, or a bundled `clawdi-runtime-sidecar/bin/clawdi-runtime-sidecar`, then starts it with `--profile-bundle`, `--proxy-url`, `--ca-file`, and optional `--secret-file` (`packages/cli/src/runtime/runtime-sidecar.ts:43`, `packages/cli/src/runtime/runtime-sidecar.ts:85`). It strips `CLAWDI_AUTH_TOKEN` and all `CLAWDI_EGRESS_*` control variables from the native sidecar process environment (`packages/cli/src/runtime/runtime-sidecar.ts:78`). The launcher waits for a JSON ready payload that returns the actual proxy URL and CA file path (`packages/cli/src/runtime/runtime-sidecar.ts:115`, `packages/cli/native/runtime-sidecar/main.go:966`).

The native sidecar itself is a loopback HTTP proxy by default: flags define `--proxy-url` with default `http://127.0.0.1:0`, a generated CA file path, an optional secret file, and an opt-in `--allow-remote-proxy` escape hatch (`packages/cli/native/runtime-sidecar/main.go:119`). It refuses to run without a profile bundle and CA file, loads enabled profiles, creates or loads the CA, listens on the configured proxy URL, and reports readiness with the actual proxy URL (`packages/cli/native/runtime-sidecar/main.go:126`, `packages/cli/native/runtime-sidecar/main.go:133`, `packages/cli/native/runtime-sidecar/main.go:145`, `packages/cli/native/runtime-sidecar/main.go:150`, `packages/cli/native/runtime-sidecar/main.go:180`).

For HTTP it expects absolute-form forward-proxy requests; otherwise it returns "this endpoint is an HTTP forward proxy" (`packages/cli/native/runtime-sidecar/main.go:551`). For HTTPS/WSS it handles `CONNECT`, writes `200 Connection Established`, terminates TLS with a minted leaf certificate from the Clawdi CA, and then forwards the inner HTTP request through the profile router (`packages/cli/native/runtime-sidecar/main.go:568`, `packages/cli/native/runtime-sidecar/main.go:585`, `packages/cli/native/runtime-sidecar/main.go:590`, `packages/cli/native/runtime-sidecar/main.go:608`). WebSocket upgrades are recognized during forwarding and routed to `forwardWebSocket`, which dials the rewritten upstream and pipes both connections after a successful `101 Switching Protocols` response (`packages/cli/native/runtime-sidecar/main.go:680`, `packages/cli/native/runtime-sidecar/main.go:731`, `packages/cli/native/runtime-sidecar/main.go:1047`).

### 1.5 Hosted runtime systemd path

Hosted runtime convergence builds a deterministic loopback egress program when the manifest has an enabled profile bundle. The proxy port is derived from instance ID and service-state path; the CA is written under `$CLAWDI_RUN_DIR/egress/systemd/ca.pem` (`packages/cli/src/runtime/manifest.ts:2214`). Clawdi writes a `clawdi-runtime-sidecar.service` user unit when either bridge surfaces or egress profiles exist; for egress it passes the profile bundle, proxy URL, CA file, and secret file to `clawdi runtime sidecar` (`packages/cli/src/runtime/manifest.ts:2881`, `packages/cli/src/runtime/manifest.ts:2895`). The `runtime sidecar` command is hosted-only, starts the bridge module if requested, starts the egress module if `CLAWDI_EGRESS_PROFILE_BUNDLE` is present, and keeps the module alive until shutdown or unexpected sidecar exit (`packages/cli/src/commands/runtime.ts:2145`, `packages/cli/src/commands/runtime.ts:2172`, `packages/cli/src/commands/runtime.ts:2185`).

This means the intended hosted architecture is process-environment transparent proxying: OpenClaw/Hermes gateway services are not wrapped with a special command, but their generated systemd environments are amended with proxy and CA variables when egress profiles exist.

## 2. Channel Profiles

### 2.1 How channel profiles are selected

Runtime channel state is projected into the desired runtime manifest before convergence. `applyRuntimeChannelsToManifestLoad` filters active channel accounts and active runtime links with agent tokens, then calls `applyRuntimeChannelProjection` (`packages/cli/src/runtime/channels.ts:63`, `packages/cli/src/runtime/channels.ts:80`). That projection always builds managed egress profiles from those links and merges them into `manifest.egressProfiles`; it also builds runtime-specific channel projections and finally applies Hermes-specific channel env settings (`packages/cli/src/runtime/channels.ts:107`, `packages/cli/src/runtime/channels.ts:111`, `packages/cli/src/runtime/channels.ts:120`, `packages/cli/src/runtime/channels.ts:127`, `packages/cli/src/runtime/channels.ts:132`).

This means the egress profile set is runtime-agnostic at the manifest layer: it is keyed by Clawdi native channel accounts and links, not by OpenClaw or Hermes. Runtime specificity enters later through OpenClaw channel config patches and Hermes environment settings, not through separate profile builders.

### 2.2 Telegram

For each active Telegram link, Clawdi creates one `http` profile. It matches `https://api.telegram.org/bot...`, requires the path to start with the linked bot token using a `secretRefPrefix`, and rewrites to `${controlPlane.apiUrl}/v1/channels/telegram` while preserving the original path (`packages/cli/src/runtime/channels.ts:313`, `packages/cli/src/runtime/channels.ts:317`, `packages/cli/src/runtime/channels.ts:319`, `packages/cli/src/runtime/channels.ts:323`, `packages/cli/src/runtime/channels.ts:331`). Telegram profile ownership is `clawdi-native-channels`, and logging redacts bot-token-shaped URL segments (`packages/cli/src/runtime/channels.ts:336`, `packages/cli/src/runtime/channels.ts:338`).

This is a transparent-proxy design for SDKs that keep the normal Telegram Bot API base host. The same projection also writes OpenClaw account config with the bot token, but no OpenClaw Telegram relay URL is required by the egress profile itself (`packages/cli/src/runtime/channels.ts:143`, `packages/cli/src/runtime/channels.ts:145`). Hermes receives `TELEGRAM_BOT_TOKEN`, `TELEGRAM_ALLOW_ALL_USERS=true`, and `HERMES_TELEGRAM_DISABLE_FALLBACK_IPS=true`, again without an explicit Clawdi Telegram base URL in this path (`packages/cli/src/runtime/channels.ts:226`).

### 2.3 Discord

For each active Discord link, Clawdi creates an `http` REST profile matching `https://discord.com/api/...` with an `Authorization: Bot <agent-token>` header check, rewriting to `${controlPlane.apiUrl}/v1/channels/discord` while preserving the path (`packages/cli/src/runtime/channels.ts:341`, `packages/cli/src/runtime/channels.ts:345`, `packages/cli/src/runtime/channels.ts:347`, `packages/cli/src/runtime/channels.ts:352`, `packages/cli/src/runtime/channels.ts:361`). It also creates a WSS `passthrough` profile for `wss://gateway.discord.gg/`, not a Clawdi gateway rewrite (`packages/cli/src/runtime/channels.ts:370`, `packages/cli/src/runtime/channels.ts:373`, `packages/cli/src/runtime/channels.ts:375`, `packages/cli/src/runtime/channels.ts:381`). Because the sidecar rejects unmatched traffic, this passthrough permits Discord gateway WebSocket traffic while REST traffic is relayed.

The distinction is important: Discord has a websocket-related profile, but it is explicit passthrough, not a managed relay profile. Tests for the native sidecar cover WSS rewrite capability in general (`packages/cli/tests/runtime-runtime-sidecar.test.ts:343`, `packages/cli/tests/runtime-runtime-sidecar.test.ts:571`), but the managed channel builder does not use that rewrite mode for Discord gateway traffic.

### 2.4 WhatsApp

For each active WhatsApp link, Clawdi creates one `http` profile for the WhatsApp Graph API. It matches `https://graph.facebook.com/v...` with a `Bearer <agent-token>` authorization header and rewrites to `${controlPlane.apiUrl}/v1/channels/whatsapp/graph` (`packages/cli/src/runtime/channels.ts:386`, `packages/cli/src/runtime/channels.ts:390`, `packages/cli/src/runtime/channels.ts:392`, `packages/cli/src/runtime/channels.ts:395`, `packages/cli/src/runtime/channels.ts:404`). There is no generated profile for Baileys WhatsApp Web WSS destinations such as `web.whatsapp.com`, `wss://web.whatsapp.com`, or `wss://...whatsapp.net` in `buildManagedChannelEgressProfiles`; the function returns after Graph profile generation (`packages/cli/src/runtime/channels.ts:413`).

Instead, Baileys routing currently appears only in runtime-specific projection data. OpenClaw receives account config with an explicit `wsUrl` pointing directly at `${controlPlane.apiUrl}/v1/channels/whatsapp/{accountId}/baileys`, plus a token and optional auth directory (`packages/cli/src/runtime/channels.ts:168`, `packages/cli/src/runtime/channels.ts:176`, `packages/cli/src/runtime/channels.ts:177`). The credential projection can materialize OpenClaw WhatsApp auth state under `.openclaw/credentials/whatsapp/<accountKey>` (`packages/cli/src/runtime/channels.ts:489`, `packages/cli/src/runtime/channels.ts:496`, `packages/cli/src/runtime/channels.ts:551`). Hermes credential projection is gated behind `HERMES_WHATSAPP_UPSTREAM_READY`, and that constant is currently false (`packages/cli/src/runtime/channels.ts:501`, `packages/cli/src/runtime/hermes-whatsapp-gate.ts:1`, `packages/cli/src/runtime/hermes-whatsapp-gate.ts:5`).

### 2.5 Runtime-agnostic profile conclusion

Channel egress profiles are runtime-agnostic relay/passthrough policy derived from active Clawdi channel links. Telegram and Discord REST are transparent relay profiles; Discord gateway is transparent passthrough; WhatsApp Graph is a transparent relay profile. WhatsApp Baileys WSS is not represented as a transparent egress profile in the current builder. Any current Baileys relay behavior must therefore come from explicit runtime plugin/gateway configuration, not from the generic egress profile path.

## 3. OpenClaw Connectors

### 3.1 Hosted image and OpenClaw code basis

The hosted runtime image does not bake a private OpenClaw payload. It copies the read-only `official-default.json` manifest into `/usr/local/share/clawdi/manifests/official-default.json` (`/home/kingsley/clawdi-hosted/infra/v2/hosted-runtime/image/Dockerfile:109`). That manifest installs OpenClaw through the official installer URL `https://openclaw.ai/install-cli.sh` and uses the official external catalog for Discord and WhatsApp plugins (`/home/kingsley/clawdi-hosted/infra/v2/hosted-runtime/image/manifests/official-default.json:21`, `/home/kingsley/clawdi-hosted/infra/v2/hosted-runtime/image/manifests/official-default.json:24`, `/home/kingsley/clawdi-hosted/infra/v2/hosted-runtime/image/manifests/official-default.json:78`, `/home/kingsley/clawdi-hosted/infra/v2/hosted-runtime/image/manifests/official-default.json:88`). The OpenClaw code checked here is the committed `release/2026.5.28` ref at `839539da6744065e61554cc38b0fd282c98c7365`.

### 3.2 Telegram

OpenClaw's Telegram default API root is the real Telegram host, `https://api.telegram.org` (`/home/kingsley/openclaw@release/2026.5.28:extensions/telegram/src/api-root.ts:1`). Clawdi's current OpenClaw channel projection writes the bot token and permissive policies, but not a Telegram relay `apiRoot` (`packages/cli/src/runtime/channels.ts:143`, `packages/cli/src/runtime/channels.ts:147`, `packages/cli/src/runtime/channels.ts:153`). That is the right shape for transparent egress: the connector keeps the provider URL, and the sidecar profile matches and rewrites `api.telegram.org`.

The OpenClaw Telegram transport honors both ordinary proxy environment variables and Clawdi's managed proxy variable. It checks for environment proxy configuration, reads `OPENCLAW_PROXY_URL`, chooses an explicit-proxy or env-proxy dispatcher, and constructs an env proxy dispatcher with `createHttp1EnvHttpProxyAgent` when applicable (`/home/kingsley/openclaw@release/2026.5.28:extensions/telegram/src/fetch.ts:238`, `/home/kingsley/openclaw@release/2026.5.28:extensions/telegram/src/fetch.ts:242`, `/home/kingsley/openclaw@release/2026.5.28:extensions/telegram/src/fetch.ts:273`, `/home/kingsley/openclaw@release/2026.5.28:extensions/telegram/src/fetch.ts:336`, `/home/kingsley/openclaw@release/2026.5.28:extensions/telegram/src/fetch.ts:347`). The runtime resolver only falls back to `OPENCLAW_PROXY_URL` if there is no explicit proxy fetch and no ambient env proxy, then computes whether to use env proxy for the Telegram API (`/home/kingsley/openclaw@release/2026.5.28:extensions/telegram/src/fetch.ts:599`, `/home/kingsley/openclaw@release/2026.5.28:extensions/telegram/src/fetch.ts:608`, `/home/kingsley/openclaw@release/2026.5.28:extensions/telegram/src/fetch.ts:609`, `/home/kingsley/openclaw@release/2026.5.28:extensions/telegram/src/fetch.ts:624`).

Conclusion: explicit Telegram `base_url` or `apiRoot` injection to Clawdi is not needed for OpenClaw. It would be redundant at best, and if pointed directly at Clawdi's relay it would bypass the provider-host match that the egress profile is designed to own.

### 3.3 Discord

For Discord REST, Clawdi currently writes the bot token and policy config, but not a Clawdi `apiBaseUrl` (`packages/cli/src/runtime/channels.ts:156`, `packages/cli/src/runtime/channels.ts:160`, `packages/cli/src/runtime/channels.ts:165`). OpenClaw's REST client defaults to `https://discord.com/api` unless an account-level base URL override exists, and only injects a custom fetch when proxy support is available (`/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/client.ts:84`, `/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/client.ts:94`, `/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/client.ts:96`, `/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/client.ts:100`). The Discord REST monitor transport explicitly builds an env proxy dispatcher from the process environment and falls back to direct only if that is unavailable (`/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/monitor/rest-fetch.ts:34`, `/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/monitor/rest-fetch.ts:37`, `/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/monitor/rest-fetch.ts:42`, `/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/monitor/rest-fetch.ts:77`, `/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/monitor/rest-fetch.ts:93`).

Conclusion for Discord REST: explicit `apiBaseUrl` injection to Clawdi is not needed. Like Telegram, the transparent architecture is provider URL plus egress env proxy plus profile rewrite.

Discord gateway is different. The managed profile is passthrough, not relay rewrite (`packages/cli/src/runtime/channels.ts:370`, `packages/cli/src/runtime/channels.ts:373`, `packages/cli/src/runtime/channels.ts:381`). OpenClaw's gateway plugin uses `ws` with a WebSocket agent only when `params.wsAgent` is set; otherwise it opens a normal WebSocket with a direct HTTPS agent (`/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/monitor/gateway-plugin.ts:170`, `/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/monitor/gateway-plugin.ts:171`, `/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/monitor/gateway-plugin.ts:173`, `/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/monitor/gateway-plugin.ts:282`). The only proxy path in this plugin comes from `discordConfig.proxy` or OpenClaw debug-proxy env; it does not read Clawdi's `OPENCLAW_PROXY_URL` (`/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/monitor/gateway-plugin.ts:272`, `/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/monitor/gateway-plugin.ts:286`, `/home/kingsley/openclaw@release/2026.5.28:extensions/discord/src/monitor/gateway-plugin.ts:291`, `/home/kingsley/openclaw@release/2026.5.28:src/proxy-capture/env.ts:98`, `/home/kingsley/openclaw@release/2026.5.28:src/proxy-capture/env.ts:103`). Therefore the Discord gateway passthrough profile only engages if OpenClaw is configured to use the sidecar proxy for gateway sockets; current Clawdi projection does not do that.

Conclusion for Discord gateway: no shim URL is needed because Clawdi is not relaying this socket, but the "transparent egress" claim is only partially true today for OpenClaw gateway traffic. To force the passthrough profile to engage, OpenClaw needs either a projected loopback `channels.discord.accounts.<id>.proxy` value or upstream support for the same ambient proxy path used by REST and WhatsApp.

### 3.4 WhatsApp Baileys

OpenClaw's WhatsApp connector can honor transparent proxying. It creates a proxy agent from ambient env for the WhatsApp WebSocket target, creates a matching fetch dispatcher for media uploads, and passes both into Baileys as `agent` and `fetchAgent` (`/home/kingsley/openclaw@release/2026.5.28:extensions/whatsapp/src/session.ts:68`, `/home/kingsley/openclaw@release/2026.5.28:extensions/whatsapp/src/session.ts:172`, `/home/kingsley/openclaw@release/2026.5.28:extensions/whatsapp/src/session.ts:173`, `/home/kingsley/openclaw@release/2026.5.28:extensions/whatsapp/src/session.ts:209`, `/home/kingsley/openclaw@release/2026.5.28:extensions/whatsapp/src/session.ts:212`). The env proxy resolver uses the standard upper/lowercase proxy variables and respects `NO_PROXY` through the proxy-agent helper (`/home/kingsley/openclaw@release/2026.5.28:extensions/whatsapp/src/session.ts:259`, `/home/kingsley/openclaw@release/2026.5.28:extensions/whatsapp/src/session.ts:263`, `/home/kingsley/openclaw@release/2026.5.28:extensions/whatsapp/src/session.ts:323`, `/home/kingsley/openclaw@release/2026.5.28:extensions/whatsapp/src/session.ts:329`). CA trust comes from Clawdi's injected Node process environment, especially `NODE_EXTRA_CA_CERTS` and `SSL_CERT_FILE`, not from OpenClaw channel-specific code (`packages/cli/src/runtime/egress-env.ts:64`).

OpenClaw also supports an explicit Baileys WebSocket override. The account config describes `wsUrl` as a Baileys-compatible proxy URL, `resolveWaWebSocketUrl` prioritizes `account.wsUrl` over `WA_WEBSOCKET_URL`, and `createWaSocket` passes it to Baileys as `waWebSocketUrl` when present (`/home/kingsley/openclaw@release/2026.5.28:extensions/whatsapp/src/accounts.ts:48`, `/home/kingsley/openclaw@release/2026.5.28:extensions/whatsapp/src/accounts.ts:54`, `/home/kingsley/openclaw@release/2026.5.28:extensions/whatsapp/src/wa-websocket-url.ts:8`, `/home/kingsley/openclaw@release/2026.5.28:extensions/whatsapp/src/wa-websocket-url.ts:59`, `/home/kingsley/openclaw@release/2026.5.28:extensions/whatsapp/src/session.ts:187`, `/home/kingsley/openclaw@release/2026.5.28:extensions/whatsapp/src/session.ts:215`). Clawdi currently populates that explicit `wsUrl` with `/v1/channels/whatsapp/{accountId}/baileys` (`packages/cli/src/runtime/channels.ts:168`, `packages/cli/src/runtime/channels.ts:176`).

Conclusion for OpenClaw WhatsApp: explicit `wsUrl` injection is not required by the OpenClaw connector's transport capabilities; it is only required by Clawdi's current missing managed Baileys WSS egress profile. In the intended transparent architecture, OpenClaw should keep Baileys' default WhatsApp Web URL and let the sidecar rewrite a websocket profile. Until that profile exists, removing `wsUrl` would stop OpenClaw Baileys relay traffic from reaching Clawdi.

## 4. Hermes Connectors

### 5.2 Overall verdict

The owner framing is correct for Telegram and Discord REST: the intended architecture is provider-native URLs plus Clawdi's local egress proxy and CA, not explicit Clawdi base URL injection.
